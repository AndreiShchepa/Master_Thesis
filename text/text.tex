In June 2016, an attacker exploited a reentrancy vulnerability in The DAO smart contract, draining approximately \$60 million worth of Ether in a matter of hours\footnote{\url{https://www.gemini.com/cryptopedia/the-dao-hack-makerdao}}. This incident, along with numerous subsequent exploits in decentralized finance (DeFi) protocols, highlights a fundamental challenge in blockchain development: once deployed, smart contracts are immutable, and vulnerabilities cannot be easily patched. Unlike traditional software where bugs might cause inconvenience or data corruption, smart contract vulnerabilities directly translate to financial losses, often measured in millions of dollars.

% The immutability of blockchain systems creates a paradox for security. While immutability is precisely what makes smart contracts trustworthy—users can verify that contract logic won't change arbitrarily—it also means that any security flaw becomes permanent. An attacker who discovers a vulnerability can exploit it with confidence, knowing the code cannot be modified to prevent the attack. Moreover, the financial nature of most smart contract applications creates strong economic incentives for exploitation. When a DeFi protocol manages hundreds of millions in assets, even sophisticated attackers invest significant resources in finding vulnerabilities. This high-stakes environment has driven the development of increasingly rigorous security audit practices, yet critical vulnerabilities continue to slip through.

Current smart contract security is highly dependent on manual code audits supplemented by automated analysis tools. Professional audit firms employ expert security researchers who spend weeks examining contract code, looking for common vulnerability patterns, reasoning about edge cases, and trying to violate protocol invariants \cite{perez2021smart}. 

% The fundamental limitation is that many critical vulnerabilities are not simple pattern matches but rather logical flaws that emerge from complex interactions between contract components or violations of implicit business logic assumptions. For example, a flash loan attack might exploit the interaction between a pricing oracle, a lending pool, and a liquidation mechanism in ways that are individually correct but collectively create arbitrage opportunities. Detecting such vulnerabilities requires understanding not just what the code does syntactically, but what it means semantically in the context of the broader protocol design. This type of reasoning has historically required human expertise, making audits time-consuming and expensive—typically taking 2-4 weeks and costing \$50,000-\$200,000 for a medium-sized protocol \cite{consensys2023audit}.

Recent advances in Large Language Models (LLMs) have demonstrated remarkable capabilities in understanding and reasoning about code. Models trained on vast corpora of source code, like GPT-4 \cite{openai2023gpt4} and Claude \cite{anthropic2024claude}, can explain code functionality, identify potential bugs, and even generate secure implementations when properly prompted. However, applying LLMs directly to smart contract security faces several challenges. First, LLMs have limited context windows—even models with 200,000 token contexts cannot process entire DeFi protocols with their complex interdependencies and imported libraries. Second, LLMs lack access to the structured analysis results produced by static analysis tools, which provide precise information about control flow, data dependencies, and detected patterns. Third, effective vulnerability analysis requires domain-specific knowledge about Solidity-specific pitfalls, common attack vectors, and audit best practices that may not be well-represented in general code training data.

This thesis addresses these challenges by developing an integrated system that combines the static analysis capabilities of the Wake framework with the reasoning abilities of LLMs through a Model Context Protocol (MCP)-based architecture. Wake is an advanced Python-based framework for Solidity analysis that provides rich programmatic access to abstract syntax trees, control flow graphs, and custom vulnerability detectors \cite{wake2023documentation}. By augmenting Wake with LLM-based analysis through MCP—a protocol that enables structured tool use by language models—we create a system where the LLM can request specific analysis results, retrieve relevant code segments, and access domain knowledge as needed, rather than requiring all context upfront.

\section{Overview}
\label{sec:overview}
Ethereum, launched in 2015, marked a significant evolution in blockchain technology. First proposed by \texttt{Vitalik Buterin} in a 2013 white paper, Ethereum extends the ledger-centric model to a full application platform that supports smart contracts and decentralized applications (dApps) \cite{Buterin2013}. Rather than simply recording value transfers, it enables general-purpose programmable logic through the EVM while preserving the core properties of transparency, decentralization,and immutability \cite{wood2014yellow}. In effect, Ethereum transformed the blockchain as a generalized state-transition machine, broadening its applicability well beyond simple currency use cases.


\subsection{Blockchain}
\label{subsec:blockchain}

To understand Ethereum's role as a programmable state-transition system, 
it is useful to begin with the blockchain foundation on which it operates. 
A blockchain is a distributed ledger maintained by a network of 
peer-to-peer nodes rather than a single central authority. It grows as an 
ordered sequence of blocks, beginning with an initial block (commonly 
called the \emph{genesis block}), where each subsequent block contains a 
complete list of transaction records and a cryptographic link to its 
predecessor. As additional blocks are appended, reversing or altering 
earlier transactions becomes computationally infeasible. Participants 
interact through addresses instead of real-world identities, enabling 
transparency while preserving pseudonymity. Because every node stores a 
copy of the chain and the full transaction history is publicly verifiable, 
the system provides strong transparency: value flows can be traced, state 
transitions inspected, and the integrity of past operations independently 
confirmed. In short, from the initial genesis block through every appended 
block, blockchain architecture combines decentralization, tamper-resistance, 
and verifiable history in a single framework \cite{blockchain}.

\begin{figure}[!htbp]
    \centering
    \includegraphics[width=0.7\linewidth]{images/block_eth.png}
    \caption{Structure of an Ethereum blockchain node \cite{blockchain_diagram_eth}.}
    \label{fig:block_eth}
\end{figure}

As illustrated in Figure~\ref{fig:block_eth}, a blockchain block links to 
its predecessor and successor by storing the cryptographic hash of the 
previous block's header within its own header, forming an immutable and 
ordered chain. Beyond this backward pointer, the block header contains 
four key fields: the \emph{previous block hash}, a \emph{timestamp} 
recording when the block was produced, a \emph{nonce} adjusted during the 
consensus process, and the \emph{Merkle root}, a single hash that 
compactly summarises all transactions contained in the block 
\cite{blockchain_diagram_eth}.

A \emph{transaction} is the fundamental unit of state change in a 
blockchain: a cryptographically signed instruction issued by a participant 
that records the transfer of value or data between addresses. Each 
transaction is broadcast to the network, validated by nodes, and 
permanently written into a block, after which it becomes immutable.

\subsubsection*{Merkle Tree}

The lower portion of Figure~\ref{fig:block_eth} expands the block's 
transaction set into a \emph{Merkle tree}: a binary tree in which each 
leaf holds the cryptographic hash of an individual transaction and each 
internal node holds the hash of its two children's concatenation. Leaf 
hashes are combined pairwise and re-hashed level by level until a single 
\emph{Merkle root} remains. Because every internal node is derived from 
its children, any alteration to a transaction propagates deterministically 
upward, changing the root and invalidating the block header --- making the 
root a binding cryptographic commitment to the entire transaction set 
\cite{merkle1987, nakamoto2008}.

This design enables inclusion proofs of size $O(\log n)$: to 
verify that a transaction belongs to a block, a verifier needs only the 
transaction itself, the header's Merkle root, and the minimal set of 
sibling hashes along the path from leaf to root --- not the full block. 
This allows lightweight validation without downloading entire blocks, 
saving bandwidth and storage. Furthermore, the previous-hash pointer in 
each header secures global block ordering: any retroactive modification 
breaks the chain link to all subsequent blocks, making tampering 
immediately detectable by any honest node in the network 
\cite{nakamoto2008, merkle_tree}.


\subsection{Consensus mechanisms}
\label{subsec:consensus}
Once the block structure is established, the network must agree on which block to append next --- a process governed by the \emph{consensus mechanism}, the design of which depends on the blockchain's trust model. Across different deployment models, two following dominant consensus families have emerged.

\subsubsection*{Proof of Work}
Proof-of-Work (PoW) is the oldest and most widely adopted consensus mechanism in blockchain networks, first introduced by Nakamoto as the backbone of Bitcoin \cite{nakamoto2008}. The fundamental principle of PoW is straightforward: nodes in a peer-to-peer network, commonly referred to as \textit{miners}, compete to solve a computationally intensive cryptographic puzzle. This puzzle requires finding a nonce value $x$ such that the hash of the block header $y = f(x)$ satisfies a predefined difficulty target --- typically that the resulting hash begins with a certain number of leading zeros. The first miner to find a valid solution broadcasts it to the network, and upon verification by other nodes, the new block is appended to the chain. The successful miner is rewarded with newly minted cryptocurrency, incentivising continued participation \cite{sabry2023}.

Each block in the chain contains the cryptographic hash of the preceding block, a timestamped list of validated transactions, and the winning nonce. This chaining structure ensures immutability: altering any historical record would require recomputing the proof-of-work for that block and every subsequent block, a feat that is computationally infeasible for any actor lacking a majority of the network's hashing power \cite{sabry2023}.

Despite its security guarantees, PoW carries a significant drawback: the competitive nature of the puzzle-solving process demands enormous and continuously growing energy consumption. As noted by Truby \cite{SCHINCKUS2021111682}, the vast majority of blockchain-based projects in the energy sector---49 out of 140 reviewed cases---relied on PoW, paradoxically consuming large amounts of electricity while simultaneously aiming to promote renewable energy adoption. This contradiction has prompted substantial debate around the environmental sustainability of PoW-based systems and has accelerated research into alternative or modified consensus protocols \cite{SCHINCKUS2021111682}.
 

\begin{figure}[!htbp]
    \centering
    \includegraphics[width=0.9\linewidth]{images/PoW.jpg}
    \caption{Proof of Work mechanism flow \cite{pow_ledger}.}
    \label{fig:pow}
\end{figure}

Figure~\ref{fig:pow} illustrates the operational flow of the PoW mechanism. A network server broadcasts a cryptographic puzzle defined as finding $x$ such that $y = f(x)$ to a set of competing miners ($x_1$, $x_2$, $x_3$). Each miner independently attempts to solve the puzzle using their available computational resources and submits their proposed solution back to the network. The server then verifies each submission: in the example shown, only the solution provided by miner $x_2$ is deemed correct, while those of $x_1$ and $x_3$ are rejected. The winning miner is subsequently awarded a token reward, reflecting the incentive structure that underpins participation in the PoW process.



\subsection{Proof of Stake}

Proof-of-Stake (PoS) emerged as a direct response to the energy and scalability limitations inherent in Proof-of-Work. Rather than requiring participants to expend computational power, PoS achieves consensus by requiring validators to lock up, or \textit{stake}, a quantity of the network's native cryptocurrency as collateral~\cite{nguyen2019pos}. The probability of a node being selected to propose and validate the next block is proportional to the size of its stake. This fundamental shift replaces the physical resource of electricity with the economic stake as the primary security guaranty of the network.

Ethereum originally operated on a PoW consensus mechanism, sharing its underlying architecture with Bitcoin. However, PoW imposed significant constraints on the network: Ethereum consumed an estimated 113 terawatt-hours of electricity annually, and its throughput was limited to approximately 15 transactions per second, which proved inadequate for the growing ecosystem of decentralized applications built on top of the platform~\cite{mit_merge}. To address these issues, the Ethereum development community pursued a long-planned transition to PoS, ultimately completed on 15 September 2022 in an event referred to as \textit{The Merge}. The transition went smoothly without major disruptions to the live network, and Ethereum's energy consumption was subsequently reduced by over 99\%\footnote{\url{https://en.wikipedia.org/wiki/Proof_of_stake}}.

Under Ethereum's current PoS implementation, a validator must stake a minimum of 32~ETH into a dedicated smart contract in order to participate in the consensus process. Block proposers are selected pseudo-randomly using a mechanism called RANDAO, and once a block is broadcast, the remaining validators attest to its validity. Finality is achieved through the Casper Friendly Finality Gadget (Casper-FFG) protocol: a block becomes finalized when two consecutive checkpoint blocks have been justified by a supermajority (two-thirds) of all staked ETH. Reverting a finalized block would require an attacker to burn at least one-third of the total staked supply, making such an attack economically devastating and effectively infeasible~\cite{ethereum_pos}.





With a solid understanding of how blocks are structured and how consensus mechanisms work, we can turn our attention to the next layer of the architecture: how transactions and smart contract code are actually executed.

\section{Ethereum Virtual Machine}
The Ethereum Virtual Machine (EVM) is a deterministic state machine that executes the bytecode of the smart contract identically on all nodes \cite{wood2014yellow}. However, it is not Turing complete in the classical sense: execution is strictly resource-bounded by gas - every instruction has a metered gas cost, and execution halts once the supplied gas is exhausted. 
This gas-bounded execution guarantees termination and prevents non-halting or deliberately expensive programs from stalling the network; accordingly, the EVM is often described as \emph{quasi–Turing complete} (computationally universal only under a finite gas budget) \cite{antonopoulos2018mastering, wood2014yellow}.

\subsection{Gas}
Gas is the unit of account for computational effort in Ethereum. Each EVM opcode has an associated gas cost, and a transaction specifies an upper bound (gas limit) on how much gas may be consumed during its execution.

Gas serves three core purposes: 
\begin{itemize}
    \item \emph{metering} computation and storage, so that the use of the resources is paid for by the initiator of a transaction.
    \item \emph{DoS resistance} and spam prevention, because excessively resource-intensive executions become economically infeasible \cite{wood2014yellow}\cite{article_1559}.
    \item \emph{economic prioritization} of limited block space via fees
\end{itemize}

Since EIP-1559\footnote{\url{https://eips.ethereum.org/EIPS/eip-1559}}, each block includes a dynamically adjusted \emph{base fee} that is burned and a user-specified \emph{priority fee} (tip) paid to the block proposer. 
Users submit \texttt{maxFeePerGas} and \texttt{maxPriorityFeePerGas}; the effective price paid per gas equals \(\min(\texttt{maxFeePerGas}, \text{base fee} + \texttt{maxPriorityFeePerGas})\). 
Blocks have an elastic gas target and the base fee increases or decreases depending on recent block gas usage, stabilizing the fee market while preserving incentives.

\subsection{Memory}

\begin{itemize}
    \item \textbf{Stack} --- low-level memory structure that operates on a last-in, first-out (LIFO) principle. It stores small local variables and value types that are required for immediate execution. The stack is limited to 1024 elements, each 256 bits wide, and is accessible only through EVM opcodes or inline assembly. Careful management is essential, as exceeding its capacity causes the execution of the contract to fail \cite{evm_storage}.
    \item \textbf{Memory} --- dynamic and expandable storage area allocated for each function execution, which is then discarded once the function has finished. Unlike the stack, which is restricted to temporary data storage, memory allows for more flexible and longer-lived data retention.
    \item \textbf{Storage} --- persistent storage is associated with every smart contract and holds its state. Read and write operations in storage are relatively more expensive than those performed in memory. The data in storage are maintained as key-value pairs \cite{evm_storage_arxiv}.
    \item \textbf{Calldata} --- read-only and immutable storage that holds function arguments and transaction data for the duration of a transaction call.
\end{itemize}

\section{Smart Contracts}
Building on the blockchain foundation introduced in Section~\ref{sec:overview}, Ethereum extends the ledger model with \emph{Smart Contracts}: programs stored on-chain and executed by all nodes under the same consensus rules \cite{wood2014yellow}. In practice, this means that the execution of the contract is deterministic and publicly verifiable: anyone can inspect the code, replay transactions, and validate the resulting state changes.

A contract is deployed through a transaction, receives its own on-chain address, and can then be called by externally owned accounts or by other contracts. This execution model reduces reliance on intermediaries and enables composable decentralized applications, where protocols interact as reusable building blocks.

\subsection{Solidity}
Solidity is Ethereum’s dominant high-level, statically typed, contract-oriented language, compiled into EVM bytecode \cite{soliditydocs}. Several language concepts are especially important from a security perspective:

\begin{itemize}
    \item \textbf{Visibility and interfaces} (\texttt{public}, \texttt{external}, \texttt{internal}, \texttt{private}) define which functions are callable and from where.
    \item \textbf{Data locations} (\texttt{storage}, \texttt{memory}, \texttt{calldata}) influence both correctness and gas usage.
    \item \textbf{External call primitives} (\texttt{call}, \texttt{delegatecall}, \texttt{staticcall}) shape trust boundaries and control flow.
    \item \textbf{Failure semantics} (\texttt{require}, \texttt{revert}, \texttt{assert}) determine how safely the contract handles unexpected states.
\end{itemize}

A common drawback of Solidity development is that subtle semantic errors can produce severe vulnerabilities. For example, performing an external call before updating the internal state or careless use of \texttt{delegatecall} may execute foreign code in the caller’s storage context.

\paragraph{Conceptual example}
\begin{minted}[fontsize=\small,breaklines]{solidity}
contract Counter {
    uint256 public value;

    function inc() external {
        value += 1; // persistent on-chain state update
    }
}
\end{minted}

Even this trivial piece of code captures the core model, where a transaction triggers EVM execution, the state is updated, and the transition becomes part of the verifiable blockchain history.

\section{Security}

In Ethereum, security is not an add-on layer, but a property that must be deliberately engineered into every line of contract code. The EVM's execution model is transparent, deterministic, and adversarially accessible by design: any deployed contract can be inspected, called, and interacted with by anyone on the network. Unlike traditional software, deployed smart contracts are typically immutable, meaning that a discovered vulnerability cannot simply be patched with a software update. The financial stakes compound this risk: contracts routinely hold millions of dollars in assets, making them high-value targets that attract sophisticated adversaries. The history of Ethereum is therefore also a history of costly security failures, many of which trace back to a small set of recurring patterns. The following sections provide a structured treatment of these patterns, grounding each in concrete code, and reference notable real-world incidents in which they were exploited to steal significant funds.

\subsection{Vulnerability Taxonomies}
%------------------------------------------------------------------

Smart contract security research and industry practice have converged on shared classification frameworks for recurring vulnerability types. Two widely used resources are the Smart Contract Weakness Classification (SWC) registry and the OWASP Smart Contract Top 10, both of which provide structured identifiers, short descriptions, and canonical examples for common issue classes~\cite{swc_registry, owasp_sc_top10}. A shared taxonomy enables consistent labeling in audit reports, static analysis tools, and vulnerability datasets. It also makes retrieval-based systems---such as the knowledge base constructed later in this thesis---easier to query and evaluate.

However, taxonomies are imperfect abstractions. Real incidents rarely map cleanly to a single label: an exploit may combine oracle manipulation with reentrancy, or rely on an access-control mistake that only becomes exploitable because of a separate accounting error. We therefore treat taxonomies as an organizational tool rather than a complete model of risk, and we emphasize actionable vulnerability \emph{patterns} that are relevant during both manual and automated code analysis~\cite{atzei2017survey, perez2021smart}.

\subsubsection*{Reentrancy attack}

This type of attack occurs when a contract makes an external call before it has finished updating its own state. The callee---which may be a contract under attacker control---can re-enter the caller inside the same transaction and observe or manipulate the intermediate, stale state~\cite{luu2016making, atzei2017survey}. The classic pattern involves a \texttt{withdraw} function that transfers Ether \emph{before} reducing the caller's recorded balance.

\begin{minted}[fontsize=\small,breaklines]{solidity}
contract VulnerableVault {
    mapping(address => uint256) public balances;

    function deposit() external payable {
        balances[msg.sender] += msg.value;
    }

    function withdraw() external {
        uint256 amount = balances[msg.sender];
        require(amount > 0, "Nothing to withdraw");

        // (1) External call -- attacker's fallback fires here
        (bool ok, ) = msg.sender.call{value: amount}("");
        require(ok, "Transfer failed");

        // (2) State update happens AFTER the call -- too late
        balances[msg.sender] = 0;
    }
}
\end{minted}

An attacker deploys a malicious contract whose \texttt{receive()} fallback immediately calls \texttt{withdraw()} again. Because \texttt{balances[msg.sender]} has not yet been set to zero at step~(1), each recursive call passes the balance check, draining the vault until gas is exhausted.

\paragraph{Attacker contract}
\begin{minted}[fontsize=\small,breaklines]{solidity}
contract Attacker {
    VulnerableVault public target;

    constructor(address _target) {
        target = VulnerableVault(_target);
    }

    function attack() external payable {
        target.deposit{value: msg.value}();
        target.withdraw();
    }

    // Re-enters withdraw() on every incoming Ether transfer
    receive() external payable {
        if (address(target).balance >= msg.value) {
            target.withdraw();
        }
    }
}
\end{minted}

The fix is straightforward: apply the \emph{checks-effects-interactions} (CEI) pattern---update all internal state \emph{before} making any external call. An additional defense is an explicit reentrancy guard\footnote{\url{TODO}}.

The most consequential reentrancy exploit in Ethereum's history occurred on 17 June 2016, when an attacker drained approximately 3.6 million ETH (valued at \$60 million at the time) from \emph{The DAO}, a decentralized autonomous investment fund that had raised \$150 million from more than 11,000 contributors~\cite{dao_hack_coindesk, atzei2017survey}. The attacker exploited a recursive split function that transferred Ether before updating the caller's token balance. Because no checks-effects-interactions discipline was applied, the attacker's malicious contract could call back into the withdrawal function repeatedly within a single transaction. The incident was so severe that it caused the Ethereum community to execute a contentious hard fork at block 1,920,000, reverting the stolen transactions---an event that permanently split the network into Ethereum (ETH) and Ethereum Classic (ETC)~\cite{dao_hack_gemini}.

\subsubsection*{Access Control}

Access control vulnerabilities arise when sensitive functions lack sufficient authorization checks, or when those checks can be circumvented due to incorrect state, incorrect use of \texttt{msg.sender}, or unsafe delegation patterns~\cite{atzei2017survey, consensys_best_practices}. A common mistake in early Solidity code was naming a constructor incorrectly (using the contract name as a function name), effectively making the ``constructor'' a public function callable by anyone after deployment.



In July 2017, an attacker exploited an access control flaw in Parity Technologies' multi-signature wallet contract to steal approximately 153,037 ETH (\$30 million at the time) from three high-profile ICO-funded projects: Aeternity, Edgeless, and Swarm City~\cite{parity_hack_openzeppelin}. The wallet contract separated its logic into a shared library using \texttt{delegatecall}, but the library's initialization function was left unprotected---callable by anyone. The attacker sent two transactions: the first claimed ownership of the multi-sig by calling the exposed initializer; the second moved all funds to an address under their control. The attack could have been prevented by not using \texttt{delegatecall} as a catch-all forwarding mechanism, and instead explicitly defining which library functions could be invoked externally.

\subsubsection*{Denial of Service}

\paragraph{Mechanism.}
Denial of service (DoS) in Ethereum manifests as making a critical operation consistently revert or prohibitively expensive. One prevalent pattern involves iterating over an unbounded array to distribute funds or update state; an attacker who controls one entry in that array can cause the iteration to fail by forcing a revert, blocking all other participants~\cite{atzei2017survey}.

\begin{minted}[fontsize=\small,breaklines]{solidity}
contract Distributor {
    address[] public recipients;

    function addRecipient(address r) external {
        recipients.push(r);
    }

    // An attacker adds a contract that always reverts on receive().

The security analysis of software systems, particularly smart contracts deployed on blockchain platforms, relies on a diverse set of analytical techniques that have evolved over decades of programming language research. This chapter provides a comprehensive overview of the foundational methods for code analysis, establishing the theoretical and practical groundwork necessary for understanding how modern tools—including Wake and its AI-augmented extensions—approach vulnerability detection in Solidity smart contracts. We organise our discussion around three complementary paradigms: static analysis, which reasons about program behaviour without execution; dynamic analysis, which observes actual runtime behaviour; and formal verification, which provides mathematical guarantees about program correctness.

\section{Static Analysis}
\label{sec:static-analysis}

Static analysis encompasses techniques that examine source code, bytecode, or other program representations to infer properties about all possible executions without actually running the program~\cite{nielson1999principles}. The appeal of static analysis lies in its soundness guarantees—a sound static analyser will never miss a bug of the class it is designed to detect—and its ability to reason about infinite execution paths. However, this power comes at the cost of precision: static analysers may report false positives, flagging code as potentially buggy when no actual vulnerability exists~\cite{rice1953classes}.


\subsection{Abstract Syntax Tree}
\label{subsec:ast}

The abstract syntax tree (AST) serves as the foundational representation for most static analysis techniques. An AST is a hierarchical tree structure that captures the syntactic structure of source code while abstracting away concrete details such as whitespace, comments, and parentheses that do not affect program semantics~\cite{aho2006compilers}. Each node in the tree corresponds to a syntactic construct—a function declaration, an assignment statement, a binary expression—and the tree's structure reflects the nesting and composition of these constructs.

    \caption{Abstract Syntax Tree example.}

\end{figure}

However, the AST has significant limitations for semantic analysis. Because it captures only syntactic structure, it cannot directly represent the flow of data through a program or the possible paths of control flow. Two syntactically similar code fragments may exhibit vastly different runtime behaviours depending on the values of variables and the paths taken through conditional statements. These limitations motivate the use of more semantically rich representations.

Intermediate representations (IRs) bridge the gap between source-level syntax and low-level execution semantics by normalizing diverse syntactic constructs into a uniform format amenable to analysis~\cite{muchnick1997advanced}. Reachability analysis over the CFG determines which program points can potentially be reached from a given starting point. \subsection{Data Flow Analysis}
\label{subsec:data-flow}

Data flow analysis tracks how values flow through a program, determining at each program point which values may be produced and consumed~\cite{kildall1973unified}. The classic formulation employs the monotone framework, which computes fixed points over the lattice of possible data flow facts by iteratively propagating information along CFG edges until no further changes occur~\cite{kam1977monotone}.

The reaching definitions analysis, a foundational data flow analysis, determines for each use of a variable which definitions (assignments) might reach that use. This information enables detection of uninitialised variables, dead code, and various forms of data dependency. For smart contract security, reaching definitions help establish whether user-controlled input can influence security-critical operations~\cite{yamaguchi2014modeling}.

Taint analysis, a specialisation of data flow analysis, tracks the flow of untrusted data from sources (such as transaction parameters) to security-sensitive sinks (such as external calls or state modifications)~\cite{schwartz2010all}. In the smart contract domain, taint sources include \texttt{msg.sender}, \texttt{msg.value}, \texttt{calldata}, and return values from external calls. Sinks include operations that transfer value, modify access control state, or perform external calls. A taint flow from source to sink without proper sanitisation indicates a potential vulnerability.

Data flow analysis faces particular challenges in Solidity due to storage aliasing. Solidity's storage model maps state variables to 256-bit storage slots according to complex packing rules, and dynamic data structures such as mappings and arrays use hash-based slot computation~\cite{antonopoulos2018mastering}. Two syntactically distinct storage references may alias to the same slot, creating data flow paths invisible to na\"ive analysis. Precise handling of storage aliasing requires modelling Solidity's storage layout rules and potentially employing alias analysis techniques~\cite{andersen1994program}.

Invariant-based fuzzing provides properties that should hold across all reachable states. The fuzzer then attempts to generate transaction sequences that violate these invariants. Common invariants include balance conservation (total value in should equal total value out), access control consistency (only authorised addresses can perform privileged operations), and state machine integrity (the contract should only be in valid states)~\cite{wustholz2020harvey}.

\section{Formal Verification}
\label{sec:formal-verification}

Formal verification employs mathematical techniques to prove that programs satisfy specified properties with certainty, not merely with high probability as testing provides~\cite{clarke2018model}. While static analysis and testing may miss vulnerabilities or report false positives, formal verification can provide guarantees: if verification succeeds, the property provably holds; if it fails, a concrete counterexample demonstrates the violation.

However, formal verification faces significant practical challenges. The techniques are computationally expensive, often requiring substantial manual effort to specify properties and guide the verification process. Scalability limitations restrict application to smaller codebases or require abstraction that may introduce imprecision. Additionally, verification establishes conformance to specification, but specifications themselves may be incomplete or incorrect, leaving gaps in security coverage.

\subsection{Symbolic Execution}
\label{subsec:symbolic-execution}

Symbolic execution executes programs with symbolic rather than concrete values, representing program state as logical constraints over symbolic variables~\cite{king1976symbolic}. Rather than computing specific outputs for specific inputs, symbolic execution computes expressions over symbolic inputs that characterise the relationship between inputs and outputs. At branch points, execution forks to explore both paths, accumulating path conditions that constrain the symbolic inputs consistent with each path.

For vulnerability detection, symbolic execution enables systematic exploration of program paths and automatic generation of inputs that trigger specific behaviours. Given a property to check—such as the absence of integer overflow—symbolic execution explores paths, collects constraints under which the property is violated, and queries a satisfiability modulo theories (SMT) solver to determine whether satisfying inputs exist. If the solver finds a satisfying assignment, it provides a concrete exploit input; if the constraints are unsatisfiable, the property holds on that path~\cite{cadar2008exe}.

Symbolic execution faces the path explosion problem: the number of paths through a program grows exponentially with the number of branch points. Loops and recursive calls exacerbate this problem, potentially creating infinite paths. Practical symbolic execution engines employ various strategies to manage path explosion, including bounded exploration, state merging, and prioritised search~\cite{cadar2013symbolic}.

In the smart contract domain, tools such as Mythril~\cite{mueller2018smashing} and Manticore~\cite{mossberg2019manticore} employ symbolic execution to detect vulnerabilities. These tools symbolically execute EVM bytecode, modelling the Ethereum execution environment including storage, message calls, and environmental variables. When a path reaches a potentially vulnerable state—an external call with user-controlled destination, an arithmetic operation that may overflow—the tool checks whether the path is feasible and, if so, reports the vulnerability with a triggering input.

Environment modelling presents particular challenges for smart contract symbolic execution. The blockchain environment includes not just the contract under analysis but the entire ecosystem of other contracts, the block context, and the transaction parameters. Accurate modelling requires assumptions about what external contracts might do, what values environmental variables might take, and how transactions might be sequenced. Conservative assumptions lead to false positives; optimistic assumptions may miss real vulnerabilities~\cite{luu2016making}.

\subsection{Theorem Proving}
\label{subsec:theorem-proving}

Theorem proving represents the most powerful—and most demanding—approach to formal verification. Rather than exhaustively exploring state space, theorem proving directly constructs mathematical proofs that programs satisfy specifications. Interactive theorem provers such as Coq~\cite{bertot2013interactive}, Isabelle~\cite{nipkow2002isabelle}, and Lean~\cite{moura2021lean} provide frameworks for developing machine-checked proofs with human guidance.

The expressiveness of theorem proving is unmatched: any mathematically statable property can, in principle, be proven. This includes properties beyond the reach of model checking and symbolic execution, such as correctness relative to complex mathematical specifications or security properties involving cryptographic assumptions. Projects such as the Ethereum Foundation's formal specification efforts demonstrate the application of theorem proving to blockchain protocol verification~\cite{hirai2017defining}.

However, the cost of theorem proving is substantial. Proofs require significant human expertise to construct, often taking person-months of effort for non-trivial systems. The proof development process requires not just programming skill but mathematical sophistication and familiarity with the proof assistant's logic and tactics. For most smart contract development contexts, this cost-benefit trade-off renders full theorem proving impractical.

Nonetheless, theorem proving plays an important role in the broader smart contract security ecosystem. Proofs of foundational libraries, such as safe arithmetic operations or standard token implementations, can be developed once and reused widely. High-value contracts managing billions of dollars may justify the investment in formal proofs. The verification of compiler correctness ensures that properties proven at the source level are preserved through compilation~\cite{leroy2009formal}.

This chapter has surveyed the foundational techniques for program analysis as applied to smart contract security. Static analysis, operating on representations from abstract syntax trees to interprocedural data flow, provides scalable vulnerability detection with soundness guaranties but may produce false positives. Dynamic analysis, through transaction tracing and fuzz testing, observes actual behaviors with precision but cannot guaranty absence of vulnerabilities. Formal verification offers mathematical certainty but at a significant computational and human cost.

Each technique occupies a distinct point in the trade-off space between soundness, completeness, scalability, and automation. In practice, effective security analysis combines multiple techniques: static analysis for broad coverage and early detection, dynamic analysis for validation and edge case discovery, and selective formal verification for highest-assurance components. This detailed defense approach acknowledges that no single technique suffices for a complete security assessment.

Despite the sophistication of these techniques, significant challenges remain. The semantic gap between source code and natural language makes it difficult for automated tools to understand programmer intent, leading to false positives (flagging intended behavior as bugs) and false negatives (missing bugs that violate unstated invariants). The evolving landscape of smart contract patterns, from novel DeFi primitives to cross-chain bridges, continuously introduces new vulnerability classes that existing detectors may not recognise~\cite{werner2022sok}.

Recent advances in large language models (LLMs) offer a promising complementary approach. LLMs trained on vast corpora of code and natural language have demonstrated remarkable capabilities in code understanding, generation, and analysis~\cite{chen2021evaluating}. Their ability to process natural language specifications, understand contextual patterns, and generalize from examples positions them as potential contributors to security analysis workflows.

\section{Sequence Probabilities}
\label{sec:seq-prob}

Let $\mathbf{x} = (x_1, x_2, \dots, x_T)$ be a sequence of tokens --- these could be words in a sentence, symbols in a formula, or any discrete units drawn from a finite vocabulary.  A \emph{language model} assigns a probability to the whole sequence by factoring it with the chain rule of probability:

\begin{equation}
    P(\mathbf{x}) \;=\; \prod_{t=1}^{T} P\!\bigl(x_t \mid x_1, \dots, x_{t-1}\bigr).
    \label{eq:chain-rule}
\end{equation}

Each factor on the right-hand side determines the likelihood of the next token given everything that preceded \cite{jm3}. Classical $n$-gram models approximate this by truncating the history to a fixed window of $n{-}1$ tokens, an assumption that works acceptably for short phrases but collapses when the prediction depends on context dozens or hundreds of tokens back. To handle longer-range dependencies, the field turned to neural networks that could, in principle, condition on the entire preceding sequence \cite{hindle2012naturalness}.


\subsection{Recurrent Neural Networks}
\label{subsec:rnn}

Recurrent Neural Networks (RNNs) offered the first neural solution to variable-length conditioning. At each time step $t$, an RNN updates a hidden state that summarises the entire prefix:

\begin{equation}
    \mathbf{h}_t = f\!\bigl(\mathbf{W}_h \mathbf{h}_{t-1} + \mathbf{W}_x \mathbf{x}_t + \mathbf{b}\bigr),
    \label{eq:rnn}
\end{equation}

where $f$ is a nonlinearity such as $\tanh$ or \footnote{\url{https://en.wikipedia.org/wiki/Rectified_linear_unit}}, and $\mathbf{W}_h$, $\mathbf{W}_x$ are learned weight matrices.  The conditional distribution for the next token is then read off from $\mathbf{h}_t$ \cite{Goodfellow-et-al-2016}.

\begin{figure}[!h]
    \centering
    \includegraphics[width=10cm, height=8cm]{images/RNN.pdf}
    \caption{Recurrent Neural Network architecture.}
    \label{fig:rnn_diagram}
\end{figure}

Figure~\ref{fig:rnn_diagram} illustrates the three main layers of an RNN and the recurrent connection that feeds the hidden state from one time step to the next.  This feedback loop allows the network to maintain a form of memory over previous inputs.  Crucially, all recurrent connections share the same weight matrix, enabling the network to process sequences of arbitrary length.

In principle, the hidden state can carry information from arbitrarily far back.  In practice, however, the repeated multiplication of weight matrices during backpropagation causes gradients to either shrink exponentially toward zero or grow without bound --- the \emph{vanishing} and \emph{exploding gradient} problems. Bengio et al.~\cite{279181} analyzed this phenomenon formally, showing that the influence of an input at time step $\tau$ on the hidden state at a later time step $t$ decays exponentially with the distance $t - \tau$.  The consequence is that standard RNNs struggle to learn dependencies that span more than roughly 10--20 time steps, even though many meaningful patterns in language and other sequential data require much longer memory.

\subsubsection*{Long Short-Term Memory}

To address this problem, Long Short-Term Memory (LSTM) networks were specifically designed~\cite{hochreiter1997long}. They introduce a dedicated \emph{cell state} that runs through the entire sequence like a conveyor belt, regulated by three learned gates: a \emph{forget gate} that decides what information to discard, an \emph{input gate} that controls what new information to store, and an \emph{output gate} that determines what part of the cell state to expose as the hidden state.  By learning to open and close these gates, LSTMs can preserve information over hundreds of time steps.

Despite this advance, a fundamental bottleneck remained: processing is strictly sequential. The token $t$ cannot be computed until the token $t{-}1$ has been processed.  This sequential dependency has two consequences.  First, it makes training on large corpora prohibitively slow, since parallelism across positions within a sequence is impossible.  Second, information about an element early in the sequence must survive passage through every intermediate hidden state before it can influence the processing of a distant element, even if the two are directly related.  Both limitations motivated the search for an architecture that could process all positions simultaneously while maintaining the ability to model long-range dependencies.

\section{Transformers}
\label{sec:transformers}
% ============================================================

The transformer, introduced by Vaswani et al.\ in 2017~\cite{vaswani2023attentionneed}, addressed both limitations of recurrent models in a single architectural innovation.  It replaced the sequential hidden-state chain with a mechanism that lets every token attend directly to every other token---\emph{self-attention}.  The result is a model that can be trained in parallel across all positions and that provides short, direct gradient paths between any pair of tokens regardless of their distance in the sequence.

The full architecture, shown in Figure~\ref{fig:transformer}, consists of an encoder--decoder stack.  The encoder maps an input sequence to a sequence of continuous representations, which the decoder then uses to generate output tokens one at a time.  However, most models relevant to contemporary NLP and code analysis use only one half of this design.  Encoder-only models such as BERT~\cite{devlin2019bert} and CodeBERT~\cite{feng2020codebert} produce contextual embeddings useful for classification and retrieval.  Decoder-only models in the GPT family~\cite{radford2019language, brown2020language} generate text autoregressively and form the backbone of today's chat-capable LLMs (GPT-5~\cite{openai2025gpt5}, Claude~\cite{anthropic2025claude}, DeepSeek~\cite{deepseekai2025deepseekv3}, Qwen~\cite{qwen2025qwen3}).  Encoder--decoder models such as T5~\cite{raffel2020exploring} and CodeT5~\cite{wang2021codet5} excel at sequence-to-sequence tasks like summarisation and translation.

The remainder of this section examines the two mechanisms that make the transformer work: self-attention, which allows tokens to exchange information regardless of distance, and positional encoding, which injects the notion of order into an otherwise position-agnostic architecture.


\subsection{Self-Attention Mechanism}
\label{subsec:self-attention}

Self-attention is the core operation that gives transformers their representational power.  The central idea is to let each token in a sequence compute a weighted combination of all other tokens, where the weights are determined by learned compatibility functions.  Concretely, each token's embedding is projected into three vectors---a \emph{query}~$\mathbf{q}$, a \emph{key}~$\mathbf{k}$, and a \emph{value}~$\mathbf{v}$---through learned linear maps:

\begin{equation}
    \mathbf{Q} = \mathbf{X}\mathbf{W}^Q, \qquad
    \mathbf{K} = \mathbf{X}\mathbf{W}^K, \qquad
    \mathbf{V} = \mathbf{X}\mathbf{W}^V,
    \label{eq:qkv}
\end{equation}

where $\mathbf{X} \in \mathbb{R}^{T \times d_{\text{model}}}$ is the matrix of input embeddings and $\mathbf{W}^Q, \mathbf{W}^K \in \mathbb{R}^{d_{\text{model}} \times d_k}$, $\mathbf{W}^V \in \mathbb{R}^{d_{\text{model}} \times d_v}$ are the projection matrices.  The attention output is then computed as:

\begin{equation}
    \text{Attention}(\mathbf{Q}, \mathbf{K}, \mathbf{V})
    \;=\; \text{softmax}\!\left(\frac{\mathbf{Q}\mathbf{K}^{\!\top}}{\sqrt{d_k}}\right)\mathbf{V}.
    \label{eq:attention}
\end{equation}

The matrix $\mathbf{Q}\mathbf{K}^{\!\top} \in \mathbb{R}^{T \times T}$ contains the compatibility score of every query--key pair.  Dividing by $\sqrt{d_k}$ is essential: without this scaling, the dot products grow in magnitude with the embedding dimension, pushing softmax into regions where its gradients are vanishingly small~\cite{vaswani2017attention}.  After softmax normalisation, each row becomes a probability distribution over positions, and the output for each token is the corresponding weighted average of value vectors.

An intuitive reading: the query at position~$t$ asks ``what do I need to know?'', the keys at all other positions advertise ``here is what I contain'', and the resulting attention weights decide how much each position contributes to the updated representation at~$t$.  This mechanism allows the model to relate any two positions in a sequence with a single computational step, regardless of their distance---a stark contrast to RNNs, where information must traverse every intermediate hidden state.


\subsubsection*{Multi-Head Attention}

Rather than computing a single attention function, transformers run $h$ independent attention heads in parallel, each with its own set of projection matrices, and concatenate the results:

\begin{equation}
    \text{MultiHead}(\mathbf{Q}, \mathbf{K}, \mathbf{V})
    = \text{Concat}\!\bigl(\text{head}_1,\;\ldots,\;\text{head}_h\bigr)\,\mathbf{W}^O,
    \label{eq:multihead}
\end{equation}
\begin{equation}
    \text{where}\quad \text{head}_i = \text{Attention}\!\bigl(\mathbf{X}\mathbf{W}^Q_i,\;\mathbf{X}\mathbf{W}^K_i,\;\mathbf{X}\mathbf{W}^V_i\bigr).
    \label{eq:head-i}
\end{equation}

The motivation for multiple heads is that a single attention function can only capture one type of relationship at a time.  By running several heads in parallel, the model can attend to different aspects of the input simultaneously.  Empirical probing studies confirm that different heads do indeed specialise in different relation types---some track syntactic dependencies, others capture coreference or semantic similarity---even without explicit supervision~\cite{clark2019does, vig2019analyzing}.  The output projection~$\mathbf{W}^O$ then combines these diverse perspectives into a single representation for each token.


\subsection{Positional Encoding}
\label{subsec:pos-enc}

Self-attention is \emph{permutation-equivariant}: if the input tokens are shuffled, the outputs are shuffled in exactly the same way, with no change in their values.  Since token order carries critical semantic information---both in natural language and in structured formats---the model must be given explicit information about position.  This is the role of \emph{positional encoding}, which is added to the input embeddings before they enter the attention layers.

The original transformer~\cite{vaswani2023attentionneed} used fixed sinusoidal functions to encode position.  For each position $\text{pos}$ and each dimension $i$ of the embedding:

\begin{equation}
    \text{PE}_{(\text{pos},\, 2i)} = \sin\!\left(\frac{\text{pos}}{10000^{2i/d_{\text{model}}}}\right), \qquad
    \text{PE}_{(\text{pos},\, 2i+1)} = \cos\!\left(\frac{\text{pos}}{10000^{2i/d_{\text{model}}}}\right).
    \label{eq:sinusoidal-pe}
\end{equation}

This design has an elegant property: the positional encoding of any fixed offset $k$ can be expressed as a linear function of the encoding at position $\text{pos}$, which in principle allows the model to learn to attend to relative positions.  However, sinusoidal encodings are fixed and cannot adapt to the data.

More recent models have moved to \emph{learned} positional encodings or, increasingly, to \emph{relative} positional encoding schemes.  Rotary Position Embedding (RoPE)~\cite{su2024roformer} encodes position by rotating query and key vectors in the complex plane, so that the dot product between a query at position $m$ and a key at position $n$ depends only on their relative distance $m - n$.  RoPE has become the dominant choice in modern decoder-only LLMs, including LLaMA~\cite{touvron2023llama} and many of its derivatives, because it combines the benefits of relative encoding with straightforward integration into the standard attention mechanism.


\subsection{Mixture-of-Experts}
\label{subsec:moe}

Standard transformers are \emph{dense}: every parameter is involved in processing every input token.  Scaling such models to hundreds of billions of parameters therefore increases compute cost proportionally.  \emph{Mixture-of-Experts} (MoE) architectures~\cite{shazeer2017outrageously} offer a way to decouple model capacity from per-token compute.  In an MoE transformer, each feed-forward sublayer is replaced by a set of $N$ parallel ``expert'' networks, and a learned \emph{router} selects only the top-$K$ experts to process each token.  The remaining experts are inactive for that token, so the model can hold far more parameters than it actually uses on any single input.

This idea has moved from a niche research topic to the dominant architecture for frontier LLMs.  The Switch Transformer~\cite{fedus2022switch} demonstrated that routing each token to a single expert could scale efficiently to trillion-parameter models.  Mixtral~\cite{jiang2024mixtral} brought MoE to open-weight models with strong performance.  Most significantly, the DeepSeek model family~\cite{deepseekai2025deepseekv3} introduced fine-grained expert segmentation---splitting experts into many smaller units and activating more of them per token---alongside shared experts that capture common knowledge across tasks, an auxiliary-loss-free load-balancing strategy, and Multi-head Latent Attention (MLA) for efficient inference.  DeepSeek-V3, with 671~billion total parameters but only 37~billion activated per token, was reportedly trained for approximately \$5.6~million---an order of magnitude less than comparable dense models---while achieving competitive performance with the best proprietary systems.

By 2025, MoE had become the architecture of choice for the majority of frontier models, adopted by DeepSeek, Mistral, Qwen, and others~\cite{nvidia2025moe}.  The key insight is that sparsity allows models to grow much larger---and thus to store more knowledge and develop more specialised internal representations---without a corresponding increase in inference cost.

With self-attention providing the ability to relate tokens across arbitrary distances, positional encoding supplying the necessary notion of order, and MoE enabling efficient scaling, the transformer has the representational machinery to model complex sequential structure at enormous scale.  The next question is how to turn this architecture into a capable model---that is, how to train it.

\section{Training Paradigms}
\label{sec:training}
% ============================================================

A transformer architecture on its own is simply a parameterised function; it becomes useful only through training.  Building a capable LLM involves three broad stages, each operating at a different scale and serving a different purpose.  \emph{Pre-training} exposes the model to vast quantities of text to learn general-purpose representations.  \emph{Fine-tuning} adapts these representations to specific tasks or desired behaviours.  \emph{Prompting} shapes the model's output at inference time through instructions and examples, without modifying any parameters.  The following subsections describe each stage in turn.


\subsection{Pre-training}
\label{subsec:pretraining}

Pre-training exposes the model to massive text corpora using self-supervised objectives that require no human labelling.  For decoder-only models, the objective is \emph{next-token prediction} (causal language modelling): given a prefix $x_1, \ldots, x_{t-1}$, maximise $\log P(x_t \mid x_1, \ldots, x_{t-1})$ summed over all positions and all documents in the corpus.  For encoder-only models like BERT~\cite{devlin2019bert}, the objective is \emph{masked language modelling}: randomly mask a fraction of tokens and train the model to recover them from bidirectional context.

The simplicity of these objectives belies the enormous computational cost of pre-training.  State-of-the-art models are trained on corpora containing trillions of tokens, using thousands of GPUs or TPUs running continuously for weeks or months.  Cottier et al.~\cite{cottier2025risingcosts} estimate that the amortised hardware and energy cost of training frontier models has grown at a rate of roughly 2.4$\times$ per year since 2016, with the largest runs already exceeding hundreds of millions of dollars and projections suggesting the billion-dollar threshold will be crossed by 2027.  The dominant cost components are AI accelerator chips and R\&D staff, each accounting for roughly a third to a half of total expenditure.  Beyond hardware, assembling and curating a high-quality training corpus is itself a major undertaking: the data must be diverse enough to cover the breadth of human knowledge, yet filtered to remove low-quality, duplicated, or toxic content.

These costs mean that pre-training a frontier dense model from scratch is feasible only for a handful of well-resourced organisations.  However, architectural innovations---particularly Mixture-of-Experts (discussed in Section~\ref{subsec:moe})---have begun to challenge this assumption: DeepSeek-V3~\cite{deepseekai2025deepseekv3} demonstrated frontier-level performance at a fraction of the cost of comparable dense models.  For most practitioners and researchers, the practical path remains to start from an existing pre-trained model and adapt it through fine-tuning---a far more accessible process that we turn to next.


\subsection{Fine-tuning}
\label{subsec:finetuning}

Fine-tuning adapts a pre-trained model to specific downstream tasks by continuing training on a smaller, curated dataset.  The most common modern approach is \emph{supervised fine-tuning} (SFT) on instruction--response pairs, which teaches the model to follow user intent reliably.  By exposing the model to examples of the desired input--output behaviour, fine-tuning can bias it toward structured outputs, improve adherence to constraints, and sharpen performance on domain-specific tasks---all without the cost of training from scratch.

However, fine-tuning is not without risks.  Training on a narrow dataset can cause \emph{catastrophic forgetting}, where the model loses general capabilities it acquired during pre-training.  Overfitting is also a concern: a model fine-tuned on a small, homogeneous dataset may perform well on similar examples but generalise poorly to novel inputs.

\textbf{Parameter-efficient methods} have made fine-tuning accessible even for large models.  Full fine-tuning updates every parameter in the model, which for billion-parameter architectures requires substantial GPU memory and risks overfitting.  LoRA (Low-Rank Adaptation)~\cite{hu2022lora} offers an elegant alternative: it freezes the pre-trained weights entirely and injects small trainable low-rank matrices into each layer, reducing the number of trainable parameters by orders of magnitude while preserving most of the model's original knowledge.  QLoRA~\cite{dettmers2023qlora} extends this further by combining LoRA with 4-bit quantisation of the frozen weights, making it possible to fine-tune models with tens of billions of parameters on a single consumer GPU.


\subsubsection*{Reinforcement Learning from Human Feedback}

Supervised fine-tuning teaches a model \emph{what} to say, but not necessarily \emph{how} to say it in a way that aligns with human preferences.  Reinforcement Learning from Human Feedback (RLHF)~\cite{ouyang2022training} addresses this gap by optimising the model's outputs to match human judgements of quality.  The process has three stages:

\begin{enumerate}
    \item \textbf{Reward model training.}  Human annotators rank multiple model outputs for the same prompt.  A separate reward model is trained on these preference pairs to predict which outputs humans would prefer.
    
    \item \textbf{Policy optimisation.}  The language model is treated as a reinforcement learning agent whose ``action'' at each step is the next token.  Using Proximal Policy Optimisation (PPO)~\cite{schulman2017proximal}, the model is fine-tuned to maximise the reward model's score while staying close to the SFT policy via a KL-divergence penalty, which prevents the model from drifting too far from coherent language.
    
    \item \textbf{Iteration.}  The process repeats: new outputs are collected, annotators provide fresh rankings, and both the reward model and the policy are updated.
\end{enumerate}

\begin{figure}[!h]
    \centering
    \includegraphics[width=9cm, height=6cm]{images/rlhf.pdf}
    \caption{Reinforcement Learning from Human Feedback pipeline.}
    \label{fig:rlhf_diagram}
\end{figure}

While RLHF has proven effective, it is complex to implement: it requires training and maintaining a separate reward model, carefully tuning the KL penalty, and managing the instability inherent in reinforcement learning.  An increasingly popular alternative is \textbf{Direct Preference Optimisation (DPO)}~\cite{rafailov2024}, which eliminates the separate reward model entirely.  DPO reformulates the RLHF objective so that the language model can be optimised directly on preference pairs---pairs of ``preferred'' and ``dispreferred'' outputs for the same prompt---using a simple classification-style loss.  This simplification reduces training complexity while achieving comparable or superior alignment quality, and has been adopted in several recent models.

\subsubsection*{Reinforcement Learning with Verifiable Rewards}

A major development in 2025 was the emergence of \emph{Reinforcement Learning with Verifiable Rewards} (RLVR), demonstrated most prominently by DeepSeek-R1~\cite{deepseekai2025deepseekr1}.  Where RLHF and DPO rely on human preference judgements, RLVR uses \emph{programmatically verifiable} signals---for example, whether generated code compiles and passes tests, or whether a mathematical derivation produces the correct answer.  DeepSeek-R1 showed that applying large-scale RL directly to a base model, using the Group Relative Policy Optimisation (GRPO) algorithm and only accuracy and format rewards, could induce sophisticated reasoning behaviours---self-reflection, verification, and dynamic strategy adaptation---without any supervised fine-tuning on human-written reasoning traces.  The resulting model achieved performance competitive with the best proprietary systems on mathematics, coding, and STEM benchmarks.

This result has broad implications.  Because verifiable rewards can be generated automatically and at scale, RLVR removes the bottleneck of expensive human annotation that constrains both RLHF and DPO.  It also demonstrated that \emph{inference-time scaling}---allowing the model to spend more computation on harder problems by generating longer chains of thought---can be a powerful complement to training-time scaling.  Reasoning models like DeepSeek-R1 and OpenAI's o-series represent a shift in how LLM capabilities are improved: rather than relying solely on larger models or better training data, substantial gains come from teaching models to \emph{think longer} at inference time~\cite{snell2024scaling}.


\subsection{Prompting}
\label{subsec:prompting}

While pre-training and fine-tuning adjust model parameters, \emph{prompting} shapes model behaviour at inference time through instructions, context, and examples---without changing any weights.  Brown et al.~\cite{brown2020languagemodelsfewshotlearners} demonstrated that including a few input--output examples (\emph{few-shot prompting}) can elicit task-specific behaviour from large decoder-only models without any fine-tuning at all.  This discovery opened the door to a rich set of techniques, collectively known as \emph{prompt engineering}, that exploit the model's in-context learning capabilities.

\textbf{Task decomposition} breaks a complex problem into smaller sub-tasks, each with narrower scope and clearer success criteria.  Rather than asking a model to solve a difficult problem in one step, the prompt guides it through a sequence of manageable steps whose outputs compose into a final answer.  This approach reduces the cognitive burden on any single inference call and makes it easier to verify intermediate results.

\textbf{Chain-of-thought (CoT)} prompting~\cite{wei2023chain} is a specific form of task decomposition in which the model is encouraged to produce intermediate reasoning steps before arriving at a final answer.  By including examples that show step-by-step reasoning, the prompt elicits similar behaviour from the model, which has been shown to improve performance on arithmetic, commonsense reasoning, and other multi-step tasks.

\textbf{Few-shot learning} calibrates what ``good'' output looks like by providing concrete examples in the prompt.  The model uses these examples to infer the desired format, style, level of detail, and reasoning strategy, then applies these inferred patterns to the actual query.  Few-shot prompting is particularly powerful when combined with retrieval: instead of relying on fixed examples, relevant demonstrations can be retrieved dynamically from a knowledge base, tailoring the prompt to each specific input.

Together, these prompting techniques allow practitioners to adapt a single general-purpose model to a wide range of tasks at inference time, complementing the more permanent adaptations achieved through fine-tuning.

\section{Embeddings}
\label{sec:embeddings}
% ============================================================

All of the approaches described so far---whether based on RNNs or transformers, whether pre-trained or fine-tuned---share a common foundation: they represent discrete symbols as continuous vectors~\cite{goodfellow2016deep}.  An \emph{embedding} is a fixed-dimensional vector representation of an input, produced by passing it through a model and extracting an intermediate or final hidden state (typically via mean pooling or a special classification token).  The key property of a good embedding is that inputs with similar meaning or function should be mapped to nearby points in vector space, enabling operations like nearest-neighbour search, clustering, and analogy reasoning to be performed directly on the representations.

The quality of embeddings depends critically on the pre-training data and objective.  A model trained exclusively on English prose will embed text according to the semantic relationships present in natural language, which may not capture the structural and functional relationships that matter in other domains.  This observation motivated the development of domain-specific embedding models, particularly for source code.


\subsection{Encoder Models}
\label{subsec:encoder-models}

General-purpose text encoders like BERT~\cite{devlin2019bert} produce strong representations for natural language tasks, but they treat source code as just another form of text, missing the structural regularities that distinguish programming languages.  Code-specialised encoder models address this gap.

\textbf{CodeBERT}~\cite{feng2020codebert} was one of the first models to pre-train a BERT-style encoder on both natural language and source code across six programming languages.  It uses both masked language modelling and replaced-token detection on NL--PL pairs, learning alignments between code and its documentation.  CodeBERT embeddings have become a standard baseline for code search, clone detection, and vulnerability classification.

\textbf{GraphCodeBERT}~\cite{guo2021graphcodebert} extends CodeBERT by incorporating data flow graphs into the pre-training objective, explicitly modelling variable definition-use chains.  This structural awareness improves performance on tasks that require understanding data dependencies---precisely the kind of reasoning needed to trace the flow of values through a program.

For smart contract vulnerability detection specifically, several studies have used encoder models as feature extractors.  Lightning Cat~\cite{lightningcat2023} uses CodeBERT preprocessing to achieve F1~scores above 93\% on the SolidiFI benchmark.  Sendner et al.~\cite{sendner2025firetries} conducted a comprehensive comparison of seven pre-trained language models---BERT, CodeBERT, GraphCodeBERT, DeBERTa, RoBERTa, DistilBERT, and Longformer---for multi-label vulnerability detection across approximately 18,000 Ethereum smart contracts.


\subsection{Seq2Seq Models}
\label{subsec:seq2seq}

\textbf{T5} (Text-to-Text Transfer Transformer)~\cite{raffel2020exploring} frames every NLP task as a text-to-text problem, enabling a single model architecture to handle classification, translation, summarisation, and generation through a unified interface.

\textbf{CodeT5}~\cite{wang2021codet5} adapts the T5 architecture for code, using an identifier-aware pre-training objective that masks and recovers code identifiers rather than random tokens.  This targeted masking forces the model to learn the roles of variables and functions within their context.  Its successor, \textbf{CodeT5+}~\cite{wang2023codet5p}, scales the model and adds contrastive learning objectives for improved representation quality.  BreachT5~\cite{breacht5_2024} demonstrated that fine-tuned CodeT5+ ensembles can perform multi-label SWC classification over large smart contract datasets, though the model's 512-token input limit constrains analysis of longer contracts.

These encoder--decoder models are particularly well-suited for tasks framed as sequence-to-sequence transformation: given a piece of code, generate a description of its behaviour; or given a natural-language specification, generate a corresponding formal property.  PropertyGPT~\cite{liu2025propertygpt} exploits this capability by using RAG over human-written verification properties to generate new formal specifications, uncovering 12~zero-day vulnerabilities that earned \$8,256 in bug bounties.

\section{LLMs in Smart Contract Auditing}
\label{sec:llm-auditing}
% ============================================================

The application of LLMs to smart contract security has evolved rapidly since 2023, with approaches spanning four broad categories: prompt-only and agentic pipelines, fine-tuned detection models, RAG-augmented frameworks, and hybrid systems that combine multiple strategies.  Each category represents a different trade-off between accessibility, precision, and the amount of domain-specific engineering required.


\subsection{Prompt-Only and Agentic Auditing Pipelines}
\label{subsec:prompt-only}

The simplest approach feeds smart contract source code directly into a general-purpose LLM with appropriate instructions.  Chen et al.~\cite{chen2024chatgpt} conducted a systematic evaluation of GPT-3.5 and GPT-4 on the SmartBugs-Curated benchmark and found high recall but poor precision, with roughly 4 out of 5 flagged issues being false positives (GPT-4 achieved 88.2\% recall but only 22.6\% precision).  Perhaps more concerning, 42\% of contracts showed unstable detection outcomes across repeated runs, highlighting the non-deterministic nature of LLM-based analysis.

More structured prompt-based approaches improve on this baseline.  \textbf{GPTLens}~\cite{hu2023gptlens} introduced an adversarial two-stage pipeline: a high-temperature ``Auditor'' agent generates vulnerability candidates, and a separate ``Critic'' agent filters false positives.  \textbf{GPTScan}~\cite{sun2024gptscan} combined GPT with static analysis for token contract auditing, achieving over 90\% precision and discovering 9~vulnerabilities missed by human auditors, at a cost of approximately \$0.01 per 1,000 lines of code.

\textbf{LLM-SmartAudit}~\cite{wei2024smartaudit} deployed multi-agent conversational frameworks with role specialisation across 6,454 contracts from Code4rena competitions.  The multi-agent design mirrors how human audit teams operate, with different agents responsible for different vulnerability categories.

These results collectively establish that general-purpose LLMs can identify security issues in smart contracts but lack the precision needed for production use without additional grounding mechanisms.


\subsection{Fine-Tuned Models for Smart Contract Vulnerability Detection}
\label{subsec:fine-tuned}

Fine-tuning domain-specific models on labelled vulnerability datasets produces substantially stronger results than prompting alone.

\textbf{iAudit}~\cite{ma2025iaudit} introduced a two-stage fine-tuning pipeline: a Detector model for binary vulnerability classification and a Reasoner model for generating human-readable explanations.  These are complemented by Ranker and Critic LLM agents that iteratively debate and refine the outputs, outperforming baselines including CodeBERT, GraphCodeBERT, CodeT5, and UniXcoder.

\textbf{Smart-LLaMA-DPO}~\cite{yu2025smartllama} applies three-stage training to LLaMA-3.1-8B: continual pre-training on smart contract code, supervised fine-tuning, then Direct Preference Optimisation using paired quality explanations.  The result surpasses GPT-4o (accuracy 58.91\%) and iAudit (accuracy 64.42\%) with improvements of +10.43\% in F1 and +7.87\% in accuracy over the previous state of the art.

Bu et al.~\cite{bu2025dapp} fine-tuned Llama3-8B on 4,998 real DApp contracts with full-parameter fine-tuning and random oversampling, reaching F1~=~0.83 and precision~0.97 for price manipulation vulnerabilities---up from F1~=~0.20 without fine-tuning.

\textbf{LLMBugScanner}~\cite{yuan2024llmbugscanner} takes an ensemble approach, combining multiple LoRA-fine-tuned models from the OpenCode leaderboard through weighted voting.  This reduces the impact of individual model weaknesses but introduces complexity in deployment.


\subsection{RAG-Augmented Detection Frameworks}
\label{subsec:rag-augmented}

Retrieval-Augmented Generation (RAG) addresses a fundamental limitation of LLMs: they cannot ``know'' about vulnerability patterns not present in their training data, and their fixed context windows may be insufficient for analysing entire protocols.  By retrieving relevant external knowledge at inference time, RAG frameworks augment the model's capabilities without retraining.

\textbf{ParaVul}~\cite{huang2025paravul} represents the most architecturally sophisticated approach.  It runs a SLoRA-fine-tuned LLaMA-13B detector and a hybrid RAG detector (combining dense embedding retrieval with BM25 lexical matching) in parallel, fusing their outputs through a meta-learning gated verification module.  The system achieves F1 scores of 0.9398 (single-label) and 0.9330 (multi-label), outperforming both standalone fine-tuning and standalone retrieval.  A final stage generates vulnerability reports using chain-of-thought prompts fed to a cloud-based LLM.

\textbf{SmartLLM}~\cite{smartllm2025} combines RAG with fine-tuned LLaMA~3.1, integrating ERC documentation for domain context.  It achieves 100\% recall with 62.5\% precision (F1~=~76.9\%), demonstrating how retrieval of standard documentation can reduce false negatives.

\textbf{PropertyGPT}~\cite{liu2025propertygpt} applies RAG in a different direction: rather than retrieving vulnerable code examples, it retrieves human-written formal verification properties from Certora audit reports and uses them to guide LLM-based property generation.  This yielded 80\% recall against ground-truth properties and uncovered 12~zero-day vulnerabilities.

\textbf{SmartGuard}~\cite{smartguard2025} retrieves semantically similar code snippets to generate chain-of-thought reasoning chains, achieving recall of 95.06\% and F1 of 94.95\% on the SolidiFI benchmark.

The most comprehensive framework is \textbf{LLM-BSCVM}~\cite{llmbscvm2025}, which implements a full vulnerability management lifecycle using a ``Decompose-Retrieve-Generate'' pipeline with multi-agent collaboration and LoRA-fine-tuned CodeLlama-13B, achieving 91\% accuracy with a 5.1\% false positive rate.

While not targeting smart contracts specifically, \textbf{ReVul-CoT}~\cite{chen2025revulcot} demonstrates the effectiveness of the RAG+CoT paradigm for general software vulnerability severity assessment.  Using retrieval from NVD and CWE databases combined with structured chain-of-thought reasoning, it achieved 87.50\% accuracy and 83.75\% Macro-F1 on 12,070 vulnerability records---improvements exceeding 10 percentage points over all baselines.  Its architecture provides a transferable blueprint for smart contract severity assessment.

\section{Internal Model}
\label{sec:wake-internal-model}

Wake constructs a rich internal representation of Solidity smart contracts that serves as the foundation for all analysis capabilities. Because of Wake build base on static analysis, all its components were described in Section \ref{sec:static-analysis}.

\subsubsection*{Intermediate Representation}
\label{subsubsec:wake-ir}

At the core of Wake's analysis capabilities lies a typed intermediate representation (IR) derived from the Solidity compiler's Abstract Syntax Tree (AST). The IR provides a structured view of the source code, representing all language constructs. The IR preserves type information and cross-references between declarations and their usages, enabling precise semantic analysis. Each IR node maintains references to its parent and children, facilitating traversal in any direction. Wake's IR is fully typed using Python's type system, ensuring type safety during analysis development.


\subsubsection*{Control Flow Graph}
\label{subsubsec:wake-cfg}

Wake constructs CFGs for functions and modifiers, representing the possible execution paths through the code. It provides utility functions for reachability analysis, such as determining whether the start node is reachable from a given position (backward reachability), or whether the success/revert end nodes are reachable from a statement (forward reachability). These primitives are essential for detecting issues like dead code or functions that always revert.

\subsubsection*{Data Dependency Graph}
\label{subsubsec:wake-ddg}

The DDG is one of Wake's most sophisticated analysis structures, tracking how data flows through the contract. It automatically expands composite types (structs, arrays, mappings) to track data flow at the member/element level. For example, when a struct is assigned, the DDG creates edges not only for the struct itself but also for each of its members recursively.

\subsubsection*{Fuzz Testing}
\label{subsubsec:wake-fuzzing}

Complementing static analysis, Wake provides robust support for dynamic testing and fuzzing:

\begin{itemize}

    \item \textbf{Property-Based Fuzzing}: Wake's fuzzer generates diverse inputs to test invariants and properties, automatically exploring edge cases that manual testing might miss.

    \item \textbf{Stateful Fuzzing}: Support for fuzzing that maintains state across operations, essential for testing complex DeFi protocols with multi-step interactions.

\subsubsection*{Fuzz Testing}

Fuzzing automatically generates large numbers of inputs to explore 
program behavior beyond what manually written tests cover. 
Property-based fuzzers such as Echidna and Foundry's fuzzer go further: 
the auditor specifies invariants that must hold under all conditions, and 
the fuzzer attempts to find any sequence of transactions that violates 
them. Stateful fuzzing maintains contract state across a sequence of 
calls, which is essential for catching vulnerabilities that only emerge 
through specific multi-step interactions---a class of bug that neither 
static analysis nor unit tests reliably detect.

The practical limitation is that the fuzzing is probabilistic rather than 
exhaustive. A bug requiring a highly specific state setup may never be 
triggered within a feasible run budget, and writing effective invariant 
specifications requires the same domain expertise as manual review. In 
practice, fuzzing therefore complements rather than replaces the earlier 
phases: static analysis provides broad fast coverage, manual review 
applies contextual reasoning, and fuzzing stress-tests the invariants 
that both preceding phases identified as critical.

\subsection{Limitations}
\label{subsec:wake-ai-limitations}

Despite its capabilities, Wake-AI faces several limitations:

\begin{itemize}
    \item \textbf{Context Window Constraints}: LLMs have limited context windows, restricting the amount of code and analysis data that can be processed simultaneously. Large codebases may require chunking strategies that could miss cross-file vulnerabilities.

    \item \textbf{Hallucination Risk}: AI models may generate plausible-sounding but incorrect vulnerability reports. The triage phase mitigates this through evidence requirements, but some false positives may persist.

    \item \textbf{Determinism}: AI model outputs are inherently non-deterministic, potentially producing different findings on repeated runs of the same codebase.

    \item \textbf{Knowledge Cutoff}: Models may lack awareness of recently discovered vulnerability patterns or new Solidity language features introduced after their training data cutoff.

    \item \textbf{Limited Tool Access}: While MCP integration provides access to Wake's analysis tools, the AI cannot execute arbitrary code or perform dynamic testing during analysis. Complex vulnerabilities requiring runtime behavior observation may be missed.
\end{itemize}

Some of these limitations merit attention and can be addressed through alternative approaches. Others, such as hallucination risk and non-determinism, cannot be fully resolved due to the inherent nature of LLMs. This thesis focuses on one of the most critical limitations: knowledge cutoff. The following chapter presents a proposed solution designed to increase the detection of genuine security issues while reducing false positives. This is achieved by providing the LLM with similar historical issues, enabling it to make more informed judgments about vulnerabilities.

\section{Design Goals}
\label{sec:design-goals}

The design of our RAG-augmented auditing system is guided by five goals, each motivated by the limitations identified in Sections~\ref{sec:llm-challenges} and~\ref{subsec:wake-ai-limitations}.

\begin{enumerate}
    \item \textbf{Groundedness.} Every claim in the LLM's output should be traceable to a concrete piece of evidence: a retrieved audit finding, a static-analysis result, or a specific code location. This directly mitigates the hallucination risk described in Section~\ref{subsec:hallucinations}. The system must present the retrieved context alongside the LLM response so that a human auditor can verify each claim.

    \item \textbf{Token efficiency.} LLM context windows are finite and costly. Rather than concatenating entire codebases or knowledge bases into the prompt, the system must retrieve a small, highly relevant subset of documents and inject only the most informative content. This keeps inference fast, reduces cost, and avoids the noise degradation observed when irrelevant material is included~\cite{lewis2020retrieval}.

    \item \textbf{Modularity.} Each component---knowledge base, embedding model, retrieval engine, re-ranker, LLM---should be independently replaceable. A modular design enables controlled ablation studies (e.g., disabling re-ranking to measure its contribution) and future upgrades (e.g., swapping embedding models or adding new data sources) without redesigning the pipeline.

    \item \textbf{Reproducibility.} Audit findings must be reproducible. The system therefore logs every configuration parameter, retrieved document, similarity score, and LLM response. Experiment snapshots capture the full settings at run time, enabling exact reproduction of any evaluation run.

    \item \textbf{Audit usefulness.} The ultimate consumers of the system's output are human auditors. The system must produce structured reports that include a verdict (vulnerable or safe), a confidence level, and a detailed reasoning trace that references specific past findings. This aligns with industry audit report conventions and allows auditors to use the output as a starting point for deeper investigation rather than a final answer.
\end{enumerate}


\section{Retrieval-Augmented Generation}
\label{sec:rag}

Retrieval-Augmented Generation (RAG), introduced by Lewis et al.~\cite{lewis2020retrieval}, addresses two fundamental limitations of standalone language models: the inability to access information beyond the training data cutoff, and the tendency to hallucinate when answering questions requiring specialised knowledge. RAG augments a parametric generator with a non-parametric retrieval component that fetches relevant documents from an external corpus at inference time.

\subsection{Formal Framework}
\label{subsec:rag-formal}

A RAG system comprises two components parameterised by $\eta$ and $\theta$ respectively. The \emph{retriever} $p_\eta$ produces a distribution over text passages~$z$ given an input query~$x$:
\begin{equation}
    p_\eta(z \mid x),
    \label{eq:retriever}
\end{equation}
where $\eta$ denotes the retriever parameters (e.g., the embedding model weights and the index structure). In practice, the retriever returns a truncated top-$k$ set of passages ranked by similarity.

The \emph{generator} $p_\theta$ produces the next output token $y_i$ conditioned on the input~$x$, a retrieved passage~$z$, and all previously generated tokens $y_{1:i-1}$:
\begin{equation}
    p_\theta(y_i \mid x, z, y_{1:i-1}),
    \label{eq:generator}
\end{equation}
where $\theta$ denotes the generator parameters (the LLM weights). The full output sequence probability marginalises over the retrieved passages:
\begin{equation}
    p(y \mid x) \;\approx\; \sum_{z \in \text{top-}k} p_\eta(z \mid x)\;\prod_{i=1}^{N} p_\theta(y_i \mid x, z, y_{1:i-1}).
    \label{eq:rag-marginal}
\end{equation}

In our setting, $x$ is a compound query consisting of the Solidity code under analysis, a description of the suspected vulnerability, and an optional vulnerability label. The retrieved passages~$z$ are historical audit findings from the knowledge base (Section~\ref{sec:knowledge-base}). The generator is GPT-4o-mini (temperature~0) that reasons about whether the suspected vulnerability truly exists, grounding its analysis in the retrieved evidence.

\subsection{Adaptation to Smart Contract Auditing}
\label{subsec:rag-adaptation}

Standard RAG architectures retrieve documents based on a single query and inject them into a single prompt. Our domain requires several adaptations:

\begin{itemize}
    \item \textbf{Multi-modal queries.} A vulnerability may be characterised by its natural-language description, by the code pattern that exhibits it, or by its category label. We support four query strategies (description, code, description-with-label, hybrid BM25) and multiple search modes (text, code, hybrid) that exploit different embedding spaces.

    \item \textbf{Structured retrieval output.} Rather than returning raw text chunks, our retriever returns rich metadata: finding title, vulnerability label, severity, summary, and code blocks. This structured output enables the prompt to present findings in a format that mirrors real audit reports.

    \item \textbf{Two-stage retrieval.} An initial broad retrieval (top-50 candidates) is followed by a cross-encoder re-ranking stage. Only the top-$k$ after re-ranking are injected into the prompt.

    \item \textbf{Hybrid retrieval.} Dense vector search excels at capturing semantic similarity but may miss lexically important terms (e.g., specific Solidity function names like \texttt{delegatecall}). BM25 keyword search complements dense retrieval by matching on exact terms. The two are merged via Reciprocal Rank Fusion (RRF)~\cite{cormack2009rrf}.
\end{itemize}

\subsection{High-Level Workflow}
\label{subsec:workflow}

The end-to-end pipeline proceeds as follows:

\begin{enumerate}
    \item \textbf{Ingestion.} New vulnerability findings are fetched from the Solodit platform via its REST API, processed (text cleaning, code block extraction, summarisation, label assignment), and stored in a PostgreSQL~17 database with the pgvector extension.

    \item \textbf{Embedding.} Each finding is embedded along two modalities: its textual summary is encoded with nomic-embed-text-v1.5 (768 dimensions), and its code blocks are encoded with CodeRankEmbed (768 dimensions). Both vectors are stored in the \texttt{finding\_embeddings} table using pgvector's \texttt{vector(768)} type.

    \item \textbf{Query formulation.} Given a piece of Solidity code and a suspected vulnerability, the system constructs a query using one of four strategies: description, code, description-with-label, or hybrid BM25.

    \item \textbf{Retrieval.} The query is embedded and used to search the vector store via PostgreSQL functions (\texttt{search\_findings\_by\_text\_ranked}, \texttt{search\_findings\_by\_code\_ranked}). In hybrid BM25 mode, dense results are fused with BM25 keyword scores via RRF.

    \item \textbf{Re-ranking.} An optional cross-encoder re-ranker (default: \texttt{cross-encoder/ms-marco-MiniLM-L-6-v2}, 22M parameters) refines the initial candidate set, improving precision before final top-$k$ selection.

    \item \textbf{Context assembly.} The top-$k$ findings are formatted into a structured context block with each finding's title, label, severity, summary, and code blocks, separated by horizontal rules.

    \item \textbf{LLM reasoning.} The assembled context, along with the Solidity code and issue description, is injected into a structured prompt. GPT-4o-mini generates a JSON response containing a verdict (\texttt{VULNERABLE} or \texttt{SAFE}), a confidence level (\texttt{HIGH}, \texttt{MEDIUM}, \texttt{LOW}), and a reasoning trace that references specific retrieved findings.

    \item \textbf{Evaluation.} Both the retrieval quality and the generation quality are evaluated against ground-truth labels and reference answers, producing quantitative metrics stored in the \texttt{evaluation\_results} table alongside the experiment configuration.
\end{enumerate}


\section{Knowledge Base}
\label{sec:knowledge-base}

The quality of a RAG system is bounded by the quality and coverage of its knowledge base. Our knowledge base contains 16{,}780 audit findings from the Solodit aggregator, producing 44{,}002 embedding vectors (16{,}780 text + 27{,}222 code).

\subsection{Data Source: Solodit}
\label{subsec:solodit}

Solodit\footnote{\url{https://solodit.cyfrin.io}} is a comprehensive aggregator of smart contract audit findings, maintained by Cyfrin. It indexes findings from major audit firms (Trail of Bits, OpenZeppelin, Spearbit, Code4rena, Sherlock, and others). Each finding includes a title, a detailed markdown content (with vulnerability description, proof of concept, recommended mitigation, and inline Solidity code), a severity classification, source metadata, and community-contributed tags.

We access Solodit through its paginated REST API. The ingestion pipeline (\texttt{run\_incremental\_ingest}) fetches findings sorted by recency, stopping upon finding an already-ingested slug (incremental ingestion). This ensures the knowledge base grows monotonically without duplication.

\paragraph{Filtering.} We restrict ingestion to findings with severity \texttt{HIGH} or \texttt{MEDIUM}. Lower-severity findings (gas optimisations, informational notes) dilute the semantic coherence of the embedding space.

\subsubsection*{Text Sanitisation}
Raw content from the API may contain null bytes that cause problems with PostgreSQL storage. The \texttt{\_sanitize\_text} function strips these before further processing. Whitespace is collapsed via regex (\texttt{re.sub(r"\textbackslash s+", " ", text).strip()}).

\subsubsection*{Code Block Extraction}
Fenced code blocks (delimited by triple backticks) are extracted using regular expression matching and stored separately in the \texttt{code\_blocks} JSONB column. This separation serves two purposes: it enables code-specific embedding using CodeRankEmbed, and it allows the textual summary to focus on the vulnerability description without code noise.

\subsubsection*{Summary Generation}
Each finding requires two summaries: a human-readable summary and a keyword-dense variant optimised for embedding retrieval (\texttt{summary\_for\_embeddings}). We support two strategies:

\begin{itemize}
    \item \textbf{LLM-based summarisation.} When enabled, GPT-4o-mini (temperature~0) receives the finding title and content (truncated to 8{,}000 characters) and produces a JSON object containing: (i)~a human-readable summary of 2--4 sentences, (ii)~a dense keyword-rich summary optimised for semantic search including vulnerability type, affected components, attack vectors, and impact, and (iii)~a vulnerability label selected from the predefined taxonomy.

    \item \textbf{Heuristic fallback.} When LLM preprocessing is disabled or fails, the system strips code blocks and markdown formatting from the content and truncates to 2{,}000 characters. The embedding summary is derived by lowercasing, removing articles (a, an, the), and collapsing whitespace.
\end{itemize}

\subsubsection*{Label Assignment}
\label{subsubsec:labeling}

Composite indexes on \texttt{(embedding\_model, content\_mode)} accelerate filtered vector queries. B-tree indexes on \texttt{custom\_label}, \texttt{impact}, \texttt{experiment\_id}, and \texttt{case\_id} support the filtering and join patterns used by the evaluation pipeline.

\paragraph{Search functions.} Four PostgreSQL functions encapsulate the similarity search logic: \texttt{search\_findings\_by\_text\_ranked} and \texttt{search\_findings\_by\_code\_ranked} accept a query vector, $k$, a similarity threshold, and an optional label filter. Similarity is computed as $1 - d_{\text{cosine}}$ using pgvector's \texttt{<=>} operator. The functions join \texttt{finding\_embeddings} with \texttt{findings}, filter by model and mode, apply the threshold, and return ranked results with full finding metadata.


\subsection{Indexing and Embeddings}
\label{subsec:indexing-embeddings}

Each finding is embedded independently along two modalities using models from the \texttt{EMBEDDING\_MODELS} registry in \texttt{config.py}.

\subsubsection*{Text Embeddings}
The textual summary is embedded using \textbf{nomic-embed-text-v1.5}~\cite{nussbaum2024nomic} (\texttt{nomic-ai/nomic-embed-text-v1.5}), a 768-dimensional model. It uses task-specific prefixes: documents are prefixed with \texttt{"search\_document: "} during indexing, and queries with \texttt{"search\_query: "} during retrieval. All embeddings are L2-normalised (\texttt{normalize\_embeddings=True}), converting cosine similarity into inner product for efficient pgvector search.

\subsubsection*{Code Embeddings}
Code blocks are embedded using \textbf{CodeRankEmbed}~\cite{nomic2024coderank} (\texttt{nomic-ai/CodeRankEmbed}), a 768-dimensional model trained for code search. It uses the prefix \texttt{"Represent this query for searching relevant code: "} for query embedding. Code embeddings enable retrieval where the query is a Solidity code snippet, directly matching code patterns across findings.

\subsubsection*{Storage}
Both models produce 768-dimensional vectors stored in the \texttt{embedding\_sm} column. The current knowledge base contains 16{,}780 text embeddings and 27{,}222 code embeddings (multiple code blocks per finding).


\section{Retrieval Pipeline}
\label{sec:retrieval}

The retrieval pipeline transforms a query into a ranked list of the $k$ most relevant historical findings. It is implemented in \texttt{SearchService} (production API) and \texttt{evaluate\_retrieval} (evaluation harness).

\subsection{Query Strategies}
\label{subsec:query-strategies}

The query strategy determines how an evaluation case is transformed into an embedding query. Four strategies are implemented in \texttt{resolve\_query}:

\begin{enumerate}
    \item \textbf{Description (\texttt{description}).} The \texttt{issue\_description} is embedded with nomic-embed-text-v1.5. The most natural strategy: the query is a natural-language vulnerability description.

    \item \textbf{Code (\texttt{code}).} The \texttt{solidity\_code} is embedded with CodeRankEmbed. Matches code structural patterns directly, bypassing the natural-language intermediary.

    \item \textbf{Description with label (\texttt{description\_label}).} The issue description is concatenated with ``\texttt{Vulnerability type: \{label\}}'' and embedded with the text model. The label narrows retrieval to a specific vulnerability class.

    \item \textbf{Hybrid BM25 (\texttt{hybrid\_bm25}).} The issue description is used for both dense vector retrieval and BM25 keyword retrieval. The two ranked lists are merged via Reciprocal Rank Fusion.
\end{enumerate}

\subsection{Dense Vector Search}
\label{subsec:vector-search}

For dense retrieval, the query is embedded and compared against stored embeddings using cosine similarity computed inside PostgreSQL. The \texttt{PgVectorStore.search} method delegates to the appropriate SQL function depending on \texttt{search\_mode}:

\begin{itemize}
    \item \texttt{search\_findings\_by\_text\_ranked} --- filters on \texttt{embedding\_model = 'nomic-text-v1.5'} and \texttt{content\_mode = 'text'}
    \item \texttt{search\_findings\_by\_code\_ranked} --- filters on \texttt{embedding\_model = 'nomic-code'} and \texttt{content\_mode = 'code'}
\end{itemize}

In \textbf{hybrid} mode (text+code), the system performs both searches independently, then merges results by \texttt{finding\_id}, keeping the higher score when a finding appears in both result sets.

\subsection{BM25 Keyword Search}
\label{subsec:bm25}

The \texttt{BM25Store} builds an in-memory BM25 index over all 16{,}780 finding summaries using the \texttt{rank-bm25} library (BM25Okapi variant). The index uses \texttt{summary\_for\_embeddings} when available, falling back to \texttt{summary}. Tokenisation is minimal: text is lowercased and split on non-alphanumeric characters, filtering out single-character tokens. This preserves domain-specific identifiers (\texttt{delegatecall}, \texttt{msg.sender}, \texttt{ERC20}) that are critical for smart contract vulnerability retrieval.

BM25 compensates for a known weakness of dense retrieval: semantic models may miss findings that use the exact Solidity identifier mentioned in the query. For example, a query about \texttt{transferFrom} reentrancy should retrieve findings mentioning \texttt{transferFrom} by name, even if their summaries use different phrasing.

\subsection{Reciprocal Rank Fusion}
\label{subsec:rrf}

When combining results from dense vectors and BM25, the scores are on different scales and cannot be directly compared. Reciprocal Rank Fusion~\cite{cormack2009rrf} merges ranked lists:
\begin{equation}
    \text{RRF}(d) = \sum_{i=1}^{L} \frac{1}{k_{\text{rrf}} + r_i(d)},
    \label{eq:rrf}
\end{equation}
where $L$ is the number of ranked lists, $r_i(d)$ is the rank of document~$d$ in list~$i$ (1-indexed), and $k_{\text{rrf}} = 60$ (default from the original paper). Documents not appearing in a given list are omitted from the sum. The implementation in \texttt{fusion.py} fetches 50 candidates from each source (configurable via \texttt{BM25Settings.bm25\_candidates}), applies RRF, and returns the fused ranking.

\subsection{Re-ranking}
\label{subsec:reranking}

Initial retrieval produces a broad candidate set optimised for recall. A re-ranker refines this set for precision. The system implements the Strategy pattern: all re-rankers implement the abstract \texttt{Reranker} base class with a \texttt{rerank(query, results, k)} method.

\subsubsection*{Cross-Encoder Re-ranking}
The \texttt{CrossEncoderReranker} processes (query, document) pairs jointly through a transformer. The default model is \texttt{cross-encoder/ms-marco-MiniLM-L-6-v2} (22M parameters), suitable for CPU inference. The system also supports:
\begin{itemize}
    \item \texttt{cross-encoder/ms-marco-MiniLM-L-12-v2} --- 33M parameters, balanced
    \item \texttt{BAAI/bge-reranker-v2-m3} --- 568M parameters, multilingual
    \item \texttt{BAAI/bge-reranker-v2-gemma} --- 2.5B parameters, most accurate
\end{itemize}

During re-ranking, each candidate's content (truncated to 4{,}096 characters) is paired with the query and scored in a single batch. Candidates are re-sorted by descending score and truncated to~$k$.

\subsubsection*{Cloud Re-ranking}
The \texttt{CohereReranker} uses the Cohere Rerank API (\texttt{rerank-v3.5}) as an alternative when GPU hardware is unavailable.

\subsubsection*{No-Op Baseline}
The \texttt{NoOpReranker} simply truncates the candidate list to~$k$ without re-scoring, enabling ablation experiments that isolate the contribution of re-ranking.


\section{Model Context Protocol Integration}
\label{sec:mcp-integration}

The Model Context Protocol (MCP)\footnote{\url{https://modelcontextprotocol.io/}} is an open standard for connecting AI models with external tools and data sources. It defines a structured interface through which an LLM-based agent can discover available tools, invoke them with typed inputs, and receive structured outputs.

\subsection{Integration with Wake-AI}
\label{subsec:mcp-wake}

Wake-AI uses MCP to expose Wake's static analysis capabilities (IR queries, CFG traversal, detector invocation) to the LLM agent. Our RAG system extends this MCP surface with additional tools for knowledge base search and retrieval:

\begin{itemize}
    \item \textbf{Search tool.} Accepts a natural-language query (and optional parameters: $k$, threshold, label filter, search mode) and returns a ranked list of similar findings with full metadata (title, summary, label, severity, code blocks, similarity score). Implemented via \texttt{SearchService.search()}.

    \item \textbf{Retrieve details tool.} Given a finding slug, returns the complete finding content, enabling the agent to request full details after an initial search returned summaries.
\end{itemize}

This design allows the LLM agent to iteratively refine its retrieval: it might first search broadly for ``reentrancy in ERC-20 transfer'', inspect the returned summaries, then retrieve full details for the most promising findings. The iterative retrieval pattern is more token-efficient than pre-loading all potentially relevant documents into the prompt.

\subsection{Security Considerations}
\label{subsec:mcp-security}

\begin{itemize}
    \item \textbf{Read-only surface.} All MCP tools are strictly read-only: they query the knowledge base but cannot modify it.

    \item \textbf{Input sanitisation.} Query strings are used as embedding inputs and BM25 tokenisation inputs. Neither pathway evaluates queries as code or SQL. The pgvector search functions use parameterised queries, preventing SQL injection.

    \item \textbf{Empty results.} The system returns an explicit ``no relevant findings found'' message rather than an empty string, preventing the LLM from hallucinating findings.

    \item \textbf{Deterministic logging.} All MCP tool invocations, inputs, and outputs are logged with timestamps for post-hoc auditing.
\end{itemize}


\section{Prompt Engineering}
\label{sec:prompt-engineering}

Prompt design governs how the LLM interprets the retrieved context and generates its analysis. Our strategy follows three principles: \emph{structured input} (clear separation of code, issue description, label, and retrieved findings), \emph{structured output} (machine-parseable JSON verdict), and \emph{evidence requirements} (explicit instruction to reference retrieved findings).

\subsection{System Prompt}
\label{subsec:system-prompt}

The system prompt (\texttt{REASONING\_SYSTEM\_PROMPT} in \texttt{runner.py}) establishes the LLM's role and output constraints. It instructs the model to:

\begin{enumerate}
    \item Examine the code for the specific vulnerability pattern described.
    \item Compare the code against retrieved past findings --- do they share the same root cause, similar constructs, or related attack vectors?
    \item Check whether protections are already in place (modifiers, require checks, safe libraries) that prevent the vulnerability.
    \item Be cautious about false positives --- only classify as \texttt{VULNERABLE} with concrete evidence.
    \item If no retrieved findings are relevant, base the verdict solely on code analysis and state this clearly.
\end{enumerate}

The required output format is a JSON object: \texttt{\{"verdict": "VULNERABLE"|"SAFE", "confidence": "HIGH"|"MEDIUM"|"LOW", "reasoning": "..."\}}.

\subsection{User Prompt}
\label{subsec:user-prompt}

The user prompt (\texttt{REASONING\_USER\_PROMPT}) assembles four sections:

\begin{enumerate}
    \item \textbf{Solidity Code Under Review} --- the code snippet in a fenced code block.
    \item \textbf{Potential Vulnerability} --- the natural-language issue description.
    \item \textbf{Vulnerability Category} --- the label (e.g., ``Reentrancy'') or ``Not assigned''.
    \item \textbf{Similar Past Audit Findings} --- the structured context from the top-$k$ retrieved findings, assembled by \texttt{\_build\_rich\_context}. Each finding is formatted with a markdown header containing title, label, and impact, followed by the finding content, with findings separated by horizontal rules.
\end{enumerate}

\subsection{File-Level Hypothesis Generation}
\label{subsec:hypothesis-generation}

For end-to-end file evaluation (\texttt{file\_eval.py}), the system first generates a vulnerability hypothesis from source code before retrieval. The hypothesis prompt (\texttt{HYPOTHESIS\_SYSTEM}) instructs the LLM to identify the most likely security vulnerability in a given Solidity function, focusing on specific patterns: reentrancy, access control, arithmetic issues, logic errors, data validation, fund loss, and front-running. The LLM responds with a concise 2--3 sentence hypothesis in plain text, which is then used as the retrieval query. This mirrors the real audit workflow: code $\rightarrow$ hypothesis $\rightarrow$ evidence retrieval $\rightarrow$ verdict.

This chapter presented the complete design of our RAG-augmented smart contract auditing methodology. The system is built around five design goals---groundedness, token efficiency, modularity, reproducibility, and audit usefulness---that address the limitations of standalone LLM-based analysis identified in earlier chapters.

The implementation comprises a knowledge base of 16{,}780 historical audit findings from Solodit (with 44{,}002 embedding vectors across two modalities), a multi-mode retrieval pipeline with BM25 fusion and cross-encoder re-ranking, MCP-based integration with Wake-AI, and structured prompting that enforces evidence-grounded JSON-format reasoning.

Experimental results are presented in Chapter~\ref{ch:results}.


\section{Research Questions}
\label{sec:research-questions}

The evaluation is structured around four research questions:

\begin{description}
    \item[RQ1: Retrieval quality.] How effectively does the retrieval pipeline surface relevant historical findings for a given vulnerability query? We measure this across query strategies (description, code, description+label, hybrid BM25), with and without re-ranking, and using both reference-based and LLM-judged metrics.

    \item[RQ2: Detection accuracy.] Does augmenting the LLM with retrieved findings improve its ability to correctly classify vulnerabilities (VULNERABLE vs.\ SAFE)? We compare RAG-augmented verdicts against ground truth on a curated golden dataset of 40 real-world findings.

    \item[RQ3: Groundedness and reasoning quality.] Are the LLM's explanations faithful to the retrieved evidence? Does the system reduce hallucinated vulnerability reports compared to ungrounded analysis?

    \item[RQ4: Component contributions.] What is the individual contribution of each pipeline component (re-ranking, BM25 fusion, self-retrieval)? Ablation experiments isolate the effect of removing or enabling each component.
\end{description}


\section{Datasets}
\label{sec:dataset}

\subsection{Knowledge Base Corpus}
\label{subsec:kb-corpus}

The knowledge base contains 16{,}780 findings from the Solodit API, filtered to HIGH and MEDIUM severity. Each finding underwent the preprocessing pipeline described in Section~\ref{subsec:preprocessing}, producing 16{,}780 text embeddings (nomic-embed-text-v1.5) and 27{,}222 code embeddings (CodeRankEmbed). The label distribution is shown in Table~\ref{tab:label-dist}.

\subsection{Evaluation Cases}
\label{subsec:eval-cases}

44 evaluation cases are stored in the \texttt{evaluation\_cases} table. Each case consists of:

\begin{itemize}
    \item \textbf{Solidity code} --- a code snippet exhibiting (or not exhibiting) a vulnerability.
    \item \textbf{Issue description} --- a natural-language description of the suspected vulnerability.
    \item \textbf{Label} --- the vulnerability category (e.g., ``Reentrancy'').
    \item \textbf{Expected verdict} --- ground-truth classification: VULNERABLE or SAFE.
    \item \textbf{Reference answer} --- an expert-written explanation serving as reference for answer correctness evaluation.
    \item \textbf{Reference finding IDs} --- UUIDs of findings known to be relevant, enabling reference-based retrieval metrics (context precision and context recall).
\end{itemize}

\subsection{Golden Dataset}
\label{subsec:golden-dataset}

A curated golden dataset of 40 real-world findings (all with \texttt{expected\_verdict = vulnerable}) is stored in \texttt{data/golden\_dataset/findings.json}. Each entry includes the finding slug (linking to the knowledge base), the vulnerable function's source code, the issue description, a reference answer, and the vulnerability label. The golden dataset spans 8 vulnerability categories and multiple projects (TraitForge, reNFT, Tapioca, Arrakis, Puffer, EigenLayer, Basin, Napier, and others). It is used for two evaluation types:

\begin{itemize}
    \item \textbf{Golden retrieval} --- given only the source code, the LLM generates a vulnerability hypothesis, which is used as the retrieval query. The target is to find the exact finding in the top-$k$ results. This measures real retrieval quality: can the system find the right vulnerability given only code, not the answer itself?

    \item \textbf{Golden generation} --- the full RAG pipeline is executed for each golden finding: retrieval, context assembly, LLM analysis, and verdict evaluation across multiple quality metrics. The finding itself can optionally be excluded from retrieved context to prevent trivial self-matching.
\end{itemize}

\subsection{File-Level Test Contract}
\label{subsec:test-contract}

A purpose-built Solidity contract (\texttt{VulnerableVault.sol}) with ground-truth labels (\texttt{VulnerableVault\_labels.json}) provides an end-to-end evaluation target. The contract contains 9 functions: 4 with known vulnerabilities and 5 that are safe, covering reentrancy, access control, arithmetic, and logic error patterns.


\section{Metrics}
\label{sec:metrics}

\subsection{Retrieval Metrics}
\label{subsec:retrieval-metrics}

\subsubsection*{Reference-Based Metrics}
Available when evaluation cases include \texttt{reference\_finding\_ids}:
\begin{itemize}
    \item \textbf{Context precision.} Fraction of retrieved findings that appear in the ground-truth relevant set:
    \begin{equation}
        \text{ContextPrecision} = \frac{|\text{retrieved} \cap \text{relevant}|}{|\text{retrieved}|}.
    \end{equation}

    \item \textbf{Context recall.} Fraction of ground-truth relevant findings that were retrieved:
    \begin{equation}
        \text{ContextRecall} = \frac{|\text{retrieved} \cap \text{relevant}|}{|\text{relevant}|}.
    \end{equation}
\end{itemize}

\subsubsection*{Label-Based Metric}
\begin{itemize}
    \item \textbf{Label match ratio.} Fraction of retrieved findings whose \texttt{custom\_label} matches the query label (case-insensitive). Returns \texttt{None} when no query label is provided:
    \begin{equation}
        \text{LabelMatch} = \frac{|\{r \in \text{retrieved} : \text{label}(r) = \text{label}(q)\}|}{|\text{retrieved}|}.
    \end{equation}
\end{itemize}

\subsubsection*{LLM-Judged Metrics}
\begin{itemize}
    \item \textbf{Context relevance (holistic).} An LLM judge evaluates the entire retrieved set, scoring on a 0--1 scale with structured feedback on coverage (\texttt{full/partial/none}) and specificity (\texttt{high/medium/low}).

    \item \textbf{Context precision (per-chunk).} Each retrieved finding is independently evaluated for binary relevance (verdict 0 or 1). The metric is the fraction of individually relevant findings. This requires one LLM call per retrieved finding but provides fine-grained diagnostics.
\end{itemize}

\subsubsection*{Golden Retrieval Metrics}
For the golden dataset (known-target evaluation):
\begin{itemize}
    \item \textbf{Hit Rate@$k$} --- binary: did the target finding appear in the top-$k$?
    \item \textbf{Mean Reciprocal Rank (MRR)} --- $\frac{1}{|Q|}\sum_{q \in Q} \frac{1}{\text{rank}(q)}$
    \item \textbf{NDCG@$k$} --- simplified for single relevant item: $\frac{1}{\log_2(\text{rank} + 1)}$
    \item \textbf{Label Hit Rate} --- fraction of queries where at least one retrieved finding shares the target's vulnerability label
\end{itemize}

\subsection{Generation Metrics}
\label{subsec:generation-metrics}

\begin{itemize}
    \item \textbf{Verdict accuracy} (deterministic). Binary comparison between predicted and expected verdict. No LLM judge required.

    \item \textbf{Faithfulness} (LLM-judged). The judge counts total factual claims in the analysis and the number supported by retrieved context, producing a grounded-claims ratio on 0--1. The response includes \texttt{claim\_count} and \texttt{supported\_count}.

    \item \textbf{Reasoning quality} (LLM-judged). Evaluates how well the analysis references past findings, draws accurate parallels, and supports its verdict. Scored 0--1 with evidence usage feedback (\texttt{strong/adequate/weak/none}).

    \item \textbf{Context utilization} (LLM-judged). Evaluates whether the analysis explicitly cites retrieved findings, draws parallels between past and current code, and logically derives the verdict from evidence. The response includes \texttt{citations\_count}, \texttt{parallels\_drawn}, and \texttt{verdict\_supported}.

    \item \textbf{Answer correctness} (LLM-judged). Compares the generated analysis against a reference answer: same verdict, same root cause, technically accurate explanation. Available only for cases with reference answers.
\end{itemize}

\subsection{Aggregate Classification Metrics}
\label{subsec:classification-metrics}

Across all evaluation cases, standard binary classification metrics are computed by treating VULNERABLE as the positive class:

\begin{itemize}
    \item \textbf{Precision}: $\text{TP} / (\text{TP} + \text{FP})$
    \item \textbf{Recall}: $\text{TP} / (\text{TP} + \text{FN})$
    \item \textbf{F1}: $2 \cdot \text{Precision} \cdot \text{Recall} / (\text{Precision} + \text{Recall})$
    \item \textbf{False positive rate}: $\text{FP} / (\text{FP} + \text{TN})$
    \item \textbf{Accuracy}: $(\text{TP} + \text{TN}) / N$
\end{itemize}

Reducing the false positive rate is a primary objective, as prior work has shown that ungrounded LLM analysis produces ${\sim}80\%$ false positives~\cite{chen2024chatgpt}.


\section{Experimental Setup}
\label{sec:experimental-setup}

\subsection{Infrastructure}
\label{subsec:infrastructure}

All experiments run on a single machine with Apple Silicon (MPS backend for sentence-transformers). PostgreSQL~17 with pgvector serves as both knowledge base and experiment store. The LLM (GPT-4o-mini, temperature~0) is accessed via the OpenAI API. Embedding models (nomic-embed-text-v1.5, CodeRankEmbed) and the cross-encoder re-ranker (ms-marco-MiniLM-L-6-v2) run locally via sentence-transformers.

\subsection{Experiment Tracking}
\label{subsec:experiment-tracking}

Each experiment run is recorded as an \texttt{Experiment} entity with a unique name, a full \texttt{config\_snapshot} (all settings serialised to JSON, excluding API keys), and a Git commit hash. All evaluation results are stored as \texttt{EvaluationResult} rows linked to the experiment, with phase, metric name, score, and structured JSON details. This design enables post-hoc comparison of arbitrary experiment pairs.

\subsection{Experiment Grid}
\label{subsec:experiment-grid}

The evaluation comprises three suites totalling 12 experiments:

\subsubsection*{Suite 1: Retrieval (8 experiments)}

\begin{itemize}
    \item \textbf{Golden retrieval} (2 experiments): LLM generates vulnerability hypothesis from code, uses it as retrieval query. Measures Hit Rate@5, MRR, NDCG@5, Label Hit Rate.
    \begin{itemize}
        \item \texttt{golden-text-k5} --- dense text search, $k=5$
        \item \texttt{golden-hybrid-k5} --- hybrid BM25, $k=5$
    \end{itemize}

    \item \textbf{Case-based retrieval} (6 experiments): Uses 44 evaluation cases with reference finding IDs. Measures context precision, recall, label match, context relevance (LLM), context precision per-chunk (LLM).
    \begin{itemize}
        \item \texttt{ret-text-k5} / \texttt{ret-text-k5-rerank} --- text mode $\pm$ cross-encoder
        \item \texttt{ret-code-k5} / \texttt{ret-code-k5-rerank} --- code mode $\pm$ cross-encoder
        \item \texttt{ret-hybrid-bm25-k5} / \texttt{ret-hybrid-bm25-k5-rerank} --- hybrid BM25 $\pm$ cross-encoder
    \end{itemize}
\end{itemize}

\subsubsection*{Suite 2: Generation (3 experiments)}

Golden generation on 40 findings. For each finding: retrieve context $\rightarrow$ generate LLM analysis $\rightarrow$ evaluate via LLM-judge (verdict accuracy, faithfulness, reasoning quality, context utilization, answer correctness).

\begin{itemize}
    \item \texttt{golden-gen-k5} --- exclude self from retrieved context (realistic scenario)
    \item \texttt{golden-gen-k5-rerank} --- exclude self + cross-encoder re-ranking
    \item \texttt{golden-gen-k5-self} --- include self in retrieved context (upper bound on what perfect retrieval would achieve)
\end{itemize}

\subsubsection*{Suite 3: End-to-End File Evaluation (1 experiment)}

\begin{itemize}
    \item \texttt{file-vault-k5} --- Full pipeline on \texttt{VulnerableVault.sol}: extract 9 functions $\rightarrow$ generate hypothesis per function $\rightarrow$ retrieve $\rightarrow$ generate verdict $\rightarrow$ compare against ground-truth labels. Measures precision, recall, F1.
\end{itemize}

Full numerical results for all 12 experiments, including per-metric breakdowns, ablation comparisons, and the Wake-AI integration audit, are presented in Chapter~\ref{ch:results}.


\section{Threats to Validity}
\label{sec:threats}

\begin{itemize}
    \item \textbf{Dataset bias.} The knowledge base is sourced exclusively from Solodit, which aggregates findings from specific audit firms. Novel vulnerability classes not covered by past audits will have no relevant findings.

    \item \textbf{Label quality.} Vulnerability labels are assigned by GPT-4o-mini. Label errors introduce noise into label-based metrics and label-filtered retrieval. The ``Logic error'' category (27\% of corpus) may be over-broad, conflating distinct vulnerability patterns.

    \item \textbf{Prompt sensitivity.} LLM outputs are sensitive to prompt phrasing. We mitigate this by using temperature~0 (greedy decoding) and fixed prompt templates, but non-determinism from API-side batching may still occur.

    \item \textbf{LLM-as-judge limitations.} Faithfulness, reasoning quality, context utilization, and answer correctness rely on GPT-4o-mini as both generator and judge. The judge may apply inconsistent scoring criteria. We mitigate this by requiring structured JSON output with specific subfields (e.g., \texttt{claim\_count}, \texttt{supported\_count} for faithfulness; \texttt{citations\_count}, \texttt{parallels\_drawn} for context utilization).

    \item \textbf{Golden dataset composition.} All 40 golden findings have \texttt{expected\_verdict = vulnerable}. The dataset contains no true negatives, so false positive rates cannot be measured from golden evaluation alone. The file-level evaluation (\texttt{VulnerableVault.sol}) compensates by including both vulnerable and safe functions.

    \item \textbf{Self-retrieval confound.} In golden retrieval, the target finding exists in the knowledge base. A hit may reflect lexical overlap between the LLM-generated hypothesis and the finding's summary rather than genuine semantic understanding. The golden generation suite with \texttt{exclude\_self=True} addresses this for the generation phase.

    \item \textbf{Reproducibility.} While all configurations and results are logged, exact reproduction depends on deterministic LLM behaviour (which OpenAI does not guarantee) and on the knowledge base state at evaluation time.
\end{itemize}
